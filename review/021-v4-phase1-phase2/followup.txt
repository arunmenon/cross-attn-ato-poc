1. BLOCKER: launcher cannot run text_only_v4

Status: FIXED.

Codex's concern: Phase 2 renamed train_lora_text_only.py →
train_text_only.py, but the launcher's arm registry was not updated.
text_only was not in ALLOWED_ARMS; lora_text in ARM_TO_TRAINER pointed
to a non-existent file. The renamed trainer also hardcoded
arm: "lora_text" in metrics output. Result: a v4 text_only_v4 config
would be rejected at validate_config; a legacy lora_text config would
pass validation but fail at trainer dispatch with "no such file"; a
manual direct invocation would record the wrong arm in metrics.json.

This is exactly the kind of "Phase 1 ships the rename, Phase 2
forgets to update the registry" gap a careful review catches. The
maintainer missed it because Phase 2's smoke test only exercised
data-pipeline code, not the launcher's arm-dispatch path.

Action: three coordinated edits.

  scripts/run_next_experiment.py (4 lines added):
    - ALLOWED_ARMS now includes both `text_only` (v4 canonical) and
      `lora_text` (v3 legacy alias).
    - ARM_SCHEMA grows a `text_only` entry that mirrors the
      lora_text shape (same exp_id/arm/training/data/eval/rationale
      keys).
    - ARM_TO_TRAINER maps BOTH text_only → train_text_only.py AND
      lora_text → train_text_only.py. The same file serves both
      contracts (v4 and v3 legacy); the trainer dispatches on the
      arm string at runtime.

  src/train/train_text_only.py:
    - The metrics row's `arm` field is now read from cfg.get("arm",
      "text_only") rather than hardcoded. Both text_only and
      lora_text round-trip correctly through metrics.jsonl. An
      unexpected arm value warns and defaults to text_only.

  src/auto_research/experiment_template.yaml:
    - The comment on the `arm:` field now lists all 6 valid values
      (xattn | cpt_light | text_only | lora_text | structured_as_text
      | event_only) with a one-line note explaining the v4 rename.

Verify (local, no GPU):

2. HIGH: structured_as_text_v4 trains under v4 prompt but evaluates under v3 prompt

Status: FIXED.

Codex's concern: the training collator dispatches on canonical fields
and calls compose_structured_as_text(row) → `<case><events>...</case>`.
But the final eval block (the one that builds eval_records before
handing them to run_three_mode_eval) unconditionally prepends
_serialize_events_compact(events) to ex["text"] →
`<events>...<case>...`. Same row, two different prompt shapes:
trained on v4, scored on v3. The structured_as_text_v4 measurement
would be invalid as a comparator against xattn_v4 (different prompt
distribution at scoring time vs training time).

This is a real bug. The fix is to apply the SAME v4-vs-v3 dispatch
in the eval-record construction as in the training collator.

Action: src/train/train_structured_as_text.py:259-280 rewritten to
replicate the v4 dispatch:
  - If row has `narrative` + `label` canonical fields → call
    compose_structured_as_text(new_ex) and assign to new_ex["text"].
    This is the v4 path; matches the training collator byte-for-byte.
  - Else → prepend _serialize_events_compact() to the existing v3
    `text` field. This is the v3 fallback path; matches legacy v3
    behavior.

The fix is small (a conditional + the import of
compose_structured_as_text) but load-bearing — without it,
structured_as_text_v4 cannot be used as the Q2 comparator the
v4 plan calls out.

Verify (local, no GPU):

    python3 -c "

    row = json.loads(open('data/samples/v4_smoke/data.jsonl').readline())

3. HIGH: phish_takeover_mfa_phished doesn't reliably include newly_added recipient

Status: FIXED, with regression assertion added.

Codex's concern: the v0 generator compared `txn_feat["rcpt_age"]` (a
raw string like "known") to `_bt("recipient_age", "known")` (a
bucket token like "<recipient_age=known>"). These literally never
match because they're different strings. The override that was
supposed to force the LAST transaction to a newly-added recipient
never fired. In Codex's 1000-seed measurement, 160 samples (16%)
of the adversarial fraud subtype had NO newly-added recipient event
at all — meaning the disambiguating cross-modal feature was missing
entirely from 16% of these samples.

This is the worst kind of bug: the docstring claims the feature is
deterministic, the test suite passed, and the data looked plausible
on inspection. But the actual generator was silently shipping
adversarial samples that lacked the adversarial feature.

Action: data/gen/journey_templates.py:gen_phish_takeover_mfa_phished.
The bug was over-engineered logic; the fix is to delete the
comparison entirely. The LAST transaction now UNCONDITIONALLY sets
rcpt_age="newly_added". This is idempotent (if the sampled value was
already newly_added, the assignment is a no-op) and makes the
adversarial-feature guarantee deterministic, not probabilistic.

Action 2 (defensive): a regression assertion added to
journey_templates._self_test(). It generates 100 samples with seeds
0..99 of `phish_takeover_mfa_phished` and asserts every single one
contains at least one event with recipient_age=newly_added. The v0
implementation fails this assertion; the fix passes it. If anyone
ever re-introduces a similar comparison bug, the self-test fires
immediately.

Verify (local, no GPU):

    python3 -c "
    import sys; sys.path.insert(0, '.')
    from data.gen.journey_templates import generate
    from data.gen.feature_bucketer import format_bucket_token as bt
    misses = []
    for s in range(1000):
        j = generate('phish_takeover_mfa_phished', s, 'human')
        if not any(ev.get('recipient_age') == bt('recipient_age', 'newly_added')
                   for ev in j.events):
            misses.append(s)
    assert len(misses) == 0, f'still missing: {len(misses)}'
    print('Finding 3 fix verified: 0/1000 misses (was 160/1000)')
    "

Result: PASS (0/1000).

4. MEDIUM: v4 contract check is only a 3-row spot check and skips v3 data silently

Status: FIXED.

Codex's concern: verify_v4_text_contract() only checks the first 3
rows (sample_n=3 in both trainers) and silently skips v3-format data.
A v4 trainer run against v3 data would print "SKIPPED" and proceed —
producing uninterpretable results because v3 data lacks the canonical
contract this measurement is designed for.

This is a fair point. The original design was "graceful degradation
for legacy data" which is the right default for diagnostic tooling
but the wrong default for v4 production trainers.

Action: two changes in data/gen/build_dataset.py:verify_v4_text_contract:

  (a) New `strict: bool = True` parameter. When strict=True, rows
      lacking canonical fields raise RuntimeError rather than skip.
      v4 production trainers (train_text_only, train_xattn) call
      with strict=True. Legacy v3 reruns can pass strict=False
      explicitly (probably via a config flag, though the trainer
      defaults to strict=True for safety).

  (b) Default sample_n raised from 3 → 32. AND the sampled indices
      are now a DETERMINISTIC SPREAD across the dataset (rows 0,
      len/N, 2*len/N, ..., len-1) rather than just the first N rows.
      This catches partial preprocessing that only mutates one part
      of the file (e.g., a script that prepends tokens to rows
      0..1000 but leaves the rest alone — invisible to a "first 3
      rows" check).

Both v4 trainer call-sites updated:
  src/train/train_text_only.py:
      verify_v4_text_contract(train_ds, sample_n=32, arm_name="text_only", strict=True)
  src/train/train_xattn.py:
      verify_v4_text_contract(train_ds, sample_n=32, arm_name="xattn", strict=True)

Note on partial scope: the v4 trainers refuse v3 data outright by
default. The Codex review's recommendation was "make canonical fields
mandatory by default, gate v3 behind an explicit legacy flag." That's
what we've done. A future legacy-mode runner that needs v3 data
should pass strict=False at the call-site (not yet wired through
config but a one-line change when needed).

Verify (local, no GPU):

    python3 -c "
    import sys; sys.path.insert(0, '.')
    from data.gen.build_dataset import verify_v4_text_contract
    v3_row = {'text': 'monolithic v3 text', 'structured_events': []}
    # strict=True: should raise
    try:
        verify_v4_text_contract([v3_row], sample_n=3, strict=True, arm_name='strict')
        assert False, 'should have raised'
    except RuntimeError as e:
        assert 'v4 contract check FAILED' in str(e)
    # strict=False: should skip with warning
    verify_v4_text_contract([v3_row], sample_n=3, strict=False, arm_name='legacy')
    print('Finding 4 fix verified')
    "

Result: PASS (strict=True hard-fails on v3 row; strict=False skips
with warning).
