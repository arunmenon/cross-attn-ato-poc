Review findings:

1. Blocker: The tie-aware metric formula rounds away the exact target.

Status: OPEN, blocker

Details:
The plan defines the core metric at .claude/tasks/baseline-evaluation-correction.md:144 as need = round(target_fpr * n_legit). That is not the exact-target metric review 018 recommended. Exact-target legit-FPR needs the fractional expected count, need = target_fpr * n_legit, then alpha = (need - n_above) / n_tied.

The difference is not cosmetic. On the current clean eval, n_legit is 2979, so target_fpr * n_legit is 29.79. The plan rounds that to 30. With the plan formula, event_only clean worst HN-FPR becomes 0.073529, lora_text becomes 0.070405, and structured_as_text becomes 0.050980. The plan's own acceptance criterion at lines 273-276 expects 0.0730, 0.0701, and 0.0507 within 1e-4, so the plan would fail its own validation.

The full-eval numbers mostly hide this because 3501 * 0.01 = 35.01 and rounding to 35 changes little. The clean-eval numbers expose the bug.

Recommendation:
Change the plan formula everywhere from round(target_fpr * n_legit) to fractional need = target_fpr * n_legit. Define threshold T as a score value where n_above <= need <= n_above + n_tied, then alpha = (need - n_above) / n_tied. Keep n_above and n_tied as integers, but keep need and alpha as floats. Update the self-test requirement to assert the clean reference values with fractional need.

2. Blocker: The plan does not apply the clean-eval mask to new launcher runs.

Status: OPEN, blocker

Details:
The plan fixes historical rows through scripts/rescore_baselines.py, but Task #37 is a new run. New runs go through scripts/run_next_experiment.py::run_post_processing, which currently invokes eval.score_risk directly on run_dir/predictions_<mode>.jsonl at lines 555-565. The trainers produce those predictions from eval_fast_path, loaded through src/train/common.py and each trainer's eval_ds_path branch. Nothing in Task 5 or Task 7 makes the normal post-run scoring path apply compute_clean_eval_mask.

That contradicts the plan objective at line 17: "update_sweep_state ranks experiments on ... computed against a leakage-filtered eval." It also confirms the maintainer's flagged worry in review/_pending-codex-plan-review.md: Task #37 would receive metric_version: 2 from the launcher while still being scored on the leaked eval.

This is exactly the kind of silent state split review 018 was trying to prevent. Re-scoring baselines is not enough; the corrected surface must be the default surface for every future x-attn row that the auto-loop ranks.

Recommendation:
Extend the plan so the primary post-processing path applies the clean eval mask for all future runs before score_risk.py and bootstrap_ci.py are called. Two acceptable designs:

1. Make run_post_processing aware of train_path/eval_fast_path, materialize filtered predictions to predictions_<mode>_clean.jsonl, and score/bootstrap those for metric_version 2.
2. Materialize a clean eval dataset directory once, update Task #37 and launcher config validation so future runs use that eval_fast_path, and record clean_eval_n/clean_eval_dropped on every new row.

Whichever route the maintainer chooses, make it explicit in Task 5 or add a new task before rescore-baselines. Add an acceptance criterion that a newly launched x-attn row is scored against 4466 clean eval rows, not just that historical baselines were re-scored.

4. Medium: The future-regeneration split plan can miss class-balance targets.

Status: OPEN, plan clarity

Details:
Task 6 assigns each unique structured_events_hash to train or eval by deterministic hash bucket before narration, then says to "Stratify within each split by journey_family for class balance" at lines 205-207. Once the split assignment is fixed by hash bucket, there is no later stratification step that can move examples across train/eval without breaking the disjointness constraint.

This does not block the immediate correction, because the current dataset is not regenerated. It does matter for the guardrail's future behavior. If duplicate-heavy skeletons are uneven by journey family, a raw hash modulo split can drift from the requested eval_frac and family balance.

Recommendation:
Change Task 6 to group rows by structured_events_hash within each journey_family and select whole groups into eval until the row-level eval_frac target is approximately met for that family. Preserve the invariant that a structured_events_hash never appears in both splits. Add build_summary fields for requested_eval_frac, actual_eval_frac, and per-family split counts.
