1. Blocker: agent_tick.sh kills the normal synchronous experiment path
   after 25 minutes.

Status: FIXED.

Codex correctly identified that my mental model of the agent tick was
wrong. AGENT_INSTRUCTIONS.md step 5 ("On completion, read...") and
step 4 ("...the launcher validates, dedups, acquires GPU lock, runs
accelerate, parses metrics, writes per-experiment results, appends to
experiments.jsonl, and refreshes sweep_state.yaml") make it explicit:
the CLI stays attached for the entire 90-min (or 150-min stress)
launcher run. My `timeout 25m` would have killed every successful
experiment at minute 25.

V7 in the brief only verified that the timeout token existed; it did
not verify that the cap was above the launcher's wall-clock cap.
That was a hole in the verification, also accepted.

Action: scripts/agent_tick.sh:183-193. Changed `timeout 25m` to
`timeout 180m`. Rationale spelled out in a 7-line block comment
above the call:

  - Normal x-attn run caps at 90 min (budget.yaml
    max_wall_clock_minutes).
  - Stress run caps at 150 min (budget.yaml
    stress_run_max_wall_clock_minutes).
  - Post-processing (bootstrap_ci on 5k records) + agent's notes
    write = a few minutes more.
  - 180 min gives ~25 min slack above the stress-run cap, which is
    enough for post-processing + notes. A truly hung CLI still dies
    in 3 hours instead of ∞.
  - GNU `timeout` default sends SIGTERM to the immediate child
    only, so the orphaned launcher continues, the trainer (in its
    own session via F6's start_new_session=True) continues, and the
    experiment completes + records correctly even if the CLI itself
    is killed mid-run. We lose the notes.md only.

Considered alternative: remove the outer cap entirely. Decided
against it because the original audit-014 F7 motivation (a wedged
CLI before launcher start would block subsequent ticks via
agent_tick.sh's GPU-lock pre-check — wait, no, lock pre-check
correctly skips when launcher started; only LOOSES a tick when
launcher hasn't started) is partially mitigated by the new F2 lock
self-heal, but a CLI that wedges in pre-launcher state (network
stall, MCP server wedge) still costs N ticks of overlap before the
operator notices. 180m bounds that worst case.

Verify (local):

    grep "timeout 180m" scripts/agent_tick.sh
    bash -n scripts/agent_tick.sh

2. Blocker: .claude/settings.json does not provide the claimed
   credential and filesystem containment.

Status: PARTIALLY FIXED. Tool-level containment now correctly
anchored and complete. OS-level containment is out of scope for this
fix pass and the F3 claim has been narrowed accordingly (audit-014
followup updated in this commit, see below).

Three sub-issues Codex raised:

(a) Path anchoring grammar wrong.
    Per https://code.claude.com/docs/en/permissions, `//path` is
    absolute, `/path` is project-root-relative. My
    `Write(/workspace/.env)` and `Write(/root/.config/anthropic/**)`
    targeted project-root-relative paths that don't exist.

    Action: .claude/settings.json now uses `//` for all absolute
    paths (`//workspace/.env`, `//workspace/.env*`, `//root/.config/
    anthropic/**`, `//root/.claude/**`, `//root/.aws/**`,
    `//root/.ssh/**`). The `**/experiments.jsonl` and similar
    project-relative globs remain as-is (those targets DO live
    inside the repo and project-root-relative is correct for them).

(b) No Read(...) denies.
    The audit-014 followup F3 claimed "denies the agent surface
    READS of the credential paths" but the actual deny list had
    only Write/Edit + a few Bash(cat ...) patterns. A non-Bash
    Read tool invocation against `//workspace/.env` would have
    been permitted.

    Action: Added 8 Read denies — `Read(.env)`,
    `Read(//workspace/.env)`, `Read(//workspace/.env*)`,
    `Read(//root/.config/anthropic/**)`, `Read(//root/.claude/**)`,
    `Read(//root/.aws/**)`, `Read(//root/.ssh/**)`,
    `Read(**/experiments.jsonl)`. Also added a few additional
    `Bash(cat //...)` doublets for the absolute-path variant of the
    pre-existing relative-path Bash denies, plus `Bash(set)` to
    cover one more env-dump path.

(c) Tool-level vs OS-level claim mis-scoping.
    Codex correctly noted that permission denies bind the
    Read/Edit/Write/Bash tool surface, not the OS. A determined Bash
    subprocess can still read a file via `python -c
    "print(open('//workspace/.env').read())"` or similar — that
    bypasses `Bash(cat ...)` denies but is still subject to the
    `Bash(...)` rule (which I have not restricted further; doing so
    would break legitimate `python3` subprocess use).

    Action: Narrowed the F3 disposition in
    review/014-auto-loop-prelaunch-audit/followup.txt: removed the
    word "enforced" and the implied OS-sandbox framing; now states
    explicitly "tool-level containment, not OS-level sandboxing".
    Added a paragraph noting that if the threat model widens beyond
    "the agent itself trying to follow conventions" to "the agent
    being adversarial or compromised", a sandboxed user + filesystem
    ACLs at the OS layer is the proper barrier. For Task #38's
    threat model (the agent is OUR code following AGENT_INSTRUCTIONS;
    the deny list is a safety net for hallucination and misclicks,
    not for a hostile agent), tool-level containment is appropriate
    but should be called what it is.

    Inline comment in .claude/settings.json's $comment block also
    spells this out, so anyone editing the file directly sees the
    framing.
