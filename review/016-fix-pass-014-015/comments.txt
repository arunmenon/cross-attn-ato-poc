Review findings:

1. Blocker: `agent_tick.sh` kills the normal synchronous experiment path after 25 minutes.

Status: OPEN, blocker

Details:
The new F7 wrapper is incompatible with the loop as documented and prompted. `scripts/agent_tick.sh:70-75` tells the agent to write a config, then run `python scripts/run_next_experiment.py ...`, and only "When it completes" read the metrics and write notes. That means a normal CLI tick is expected to stay alive while the launcher runs the trainer. The same file's comments describe 90-minute experiments (`scripts/agent_tick.sh:13`), and `scripts/run_next_experiment.py:446` still gives normal experiments a 90-minute wall-clock cap.

But `scripts/agent_tick.sh:182` wraps the entire CLI invocation in `timeout 25m "$CLI"`. A successful 90-minute x-attn run is therefore treated as a hung CLI at minute 25. Depending on how the Claude CLI and GNU timeout propagate SIGTERM to child tool processes on the pod, the failure mode is either: kill/interrupt the launcher mid-experiment, orphan the launcher/trainer without the agent, or at minimum prevent the agent from completing the post-run notes/readback phase it was explicitly instructed to do. None of those is safe for unattended cron.

This is not caught by the V7 source probe because the probe only asserts that `timeout 25m` exists. The problem is the lifecycle: the timeout is shorter than the work it wraps.

Recommendation:
Do not activate cron with this wrapper. Either remove the outer 25-minute timeout, make the agent submit the launcher in a deliberately detached/supervised way and exit quickly, or set the CLI timeout above the maximum launcher wall-clock plus post-processing time and verify signal propagation. The key invariant is that the timeout must not fire during a normal `run_next_experiment.py` run.

2. Blocker: `.claude/settings.json` does not provide the claimed credential and filesystem containment.

Status: OPEN, blocker

Details:
The fix creates a real `.claude/settings.json`, and Claude Code 2.1.142 supports project settings and `permissions.deny`. The specific rules in this file are still not enough to close audit-014 F3.

First, the credential path rules are mis-anchored for Claude Code's documented path grammar. The docs say `//path` is an absolute filesystem path, while `/path` is relative to the project root. Current rules such as `Write(/workspace/.env)` and `Write(/root/.config/anthropic/**)` therefore do not target `/workspace/.env` or `/root/.config/anthropic/**` as absolute paths.

Second, there are no `Read(...)` deny rules at all. The followup says the agent is denied read access to `/workspace/.env` and `/root/.config/anthropic/**`, but the actual deny list only has Write/Edit rules plus a few Bash patterns (`Bash(cat ...)`, `Bash(env)`, `Bash(printenv)`). Official Claude Code docs recommend `Read(...)` denies for sensitive files and note that permission rules are not OS-level sandboxing for arbitrary subprocess file access. With no `Read(//workspace/.env*)`, no `Read(//root/.config/anthropic/**)`, no sandbox section, and no active pod-side deny test yet, the credential-read side of F3 remains open.

Third, the same limitation applies to state/source containment through Bash. Write/Edit denies protect Claude's file-editing tools, but they are not a hard filesystem boundary for a Bash subprocess that writes files itself. That may be acceptable if the intended barrier is "tool-level deny plus convention", but it is not the enforced filesystem containment claimed by audit-014 F3.

Recommendation:
Treat F3 as still open. Add correctly anchored `Read/Edit/Write` denies for absolute credential paths using `//...`, add explicit `Read` denies for secrets, and either enable an appropriate sandbox / hook / managed policy or narrow the claim to tool-level containment. Then verify on the pod with an actual denied operation before starting cron.

Bottom line:
F1, F2, F4, F5, F6, and the AGENT_INSTRUCTIONS HN-FPR migration are in good shape. The cron loop is still blocked by the F7 timeout lifecycle bug and incomplete F3 containment. Fix those before Task #38 activation.
