Review findings:

1. High: LLM narration receives and can leak semantic actor labels.

Status: OPEN, leakage risk before LLM-mode Day-1 generation

Details:
data/gen/narrative_generator.py lines 185-204 serialize the LLM prompt, and
line 192 emits `Actor type: {actor}.`. For fraud-heavy actor classes this is
not a neutral structural cue; it is semantic label material:
`agent_compromised`, `agent_adversarial`, and even `agent_buying` are exposed
directly to the narrator. The scan layer then fails to catch this class of leak.
I verified that `narrative_leakage_scan('The compromised agent completed the
session.')` and `narrative_leakage_scan('An adversarial agent attempted
automation.')` both return clean. This is exactly the kind of token/label
leakage the three-mode eval regime is supposed to prevent: `<actor_*>` tags are
stripped or opacified at eval, but leaked actor semantics in the narrative body
would remain visible to every text baseline.

Recommendation:
Do not pass raw actor-family names to the LLM. Replace `Actor type:
agent_compromised` style input with neutral cadence/tool-use descriptors, e.g.
"interaction cadence: highly regular", "tool-use trace: present", "actor
surface: agent-mediated" without compromised/adversarial/buying/finance class
names. Also extend `narrative_leakage_scan` and the narrator prompt ban list to
catch actor-family semantic labels that should not appear in narrative text:
compromised, adversarial, malicious agent, buying assistant, finance assistant,
hybrid actor if you intend those to be stripped/opacified rather than visible.

2. High: PII boundary assertion misses raw synthetic IDs in normal PII fields.

Status: OPEN, PII fencing guardrail gap

Details:
data/gen/pii_fencer.py lines 129-152 only reject a short list of `*_real`
keys, email-like strings, and IPv4-like strings. It does not enforce that
identifier fields such as `recipient`, `merchant`, `device_id`, `acct_id`,
`email`, `phone`, `ip`, and `browser` contain the approved fence tokens. I
verified that these raw event values pass undetected:
    {'recipient': 'recipient_abc123'}
    {'device_id': 'device_deadbeef'}
    {'acct_id': 'acct_123456'}

This weakens the "last-line defense" that build_dataset.journey_to_record
claims to run before writing. If a future generator accidentally assigns a raw
synthetic ID to a normal PII key instead of using MetadataKeeper, the build will
not abort.

Recommendation:
Make `assert_no_raw_pii_in_event` key-aware. For PII-bearing keys, require the
value to equal the corresponding fence token exactly:
    recipient -> <recipient>
    merchant -> <merchant>
    ip -> <ip>
    device_id -> <device_id>
    acct_id -> <acct_id>
    email -> <email>
    phone -> <phone>
    browser -> <browser>
Keep the existing string-pattern checks as defense in depth, but do not rely on
them for structured event fields. Add negative self-tests for each PII key.

3. Medium: CostTracker can accept and cache a narrative after exceeding budget.

Status: OPEN, cost guardrail gap

Details:
docs/batch-2-data-generators.md line 150 says `CostTracker.over_budget()` is
checked before every call, and PLAN.md line 589 says the LLM narration path is
capped. In data/gen/narrative_generator.py lines 283-287 the budget is checked
only before the call. Lines 299-308 then charge the actual usage and return the
body if the leakage scan is clean, even if that charge took the tracker over
budget. A synthetic narrator with high token counts produced:
    {'spent_usd': 4.8, 'budget_usd': 0.5, 'n_calls': 1, ...}
and `generate_narrative` still returned normally.

For the normal Haiku path, overshoot may be only one call, but the current
implementation is not a hard cap and can be much worse when the usage estimate
or injected narrator is wrong.

Recommendation:
After `tracker.charge(...)`, check whether `tracker.spent_usd >
tracker.budget_usd`. If it does, fail loudly before caching or returning the
narrative. If you want to avoid making an unrecoverable API call when budget is
nearly exhausted, add a pre-call estimated-cost guard based on serialized prompt
length and `max_tokens`, but the post-charge guard is the minimum fix.

4. Medium: LLM retry loop is likely to repeat the same leaky output.

Status: OPEN, LLM-mode robustness gap

Details:
data/gen/narrative_generator.py lines 297-311 loop over retries but call the
same narrator with the same journey and prompt each time. The code does not vary
the prompt with the previous leakage hits, does not pass any retry-specific
instruction, and does not set a temperature. If the provider output is
deterministic or sticky for this prompt, a leakage failure will repeat until the
retry budget is exhausted. I verified this behavior with an injected leaky
narrator: the function made three attempts and failed with the same hit class.
That is a loud failure rather than silent leakage, which is good, but it means
Day-1 LLM generation can drop or fail rows that a targeted retry prompt could
recover.

Recommendation:
On retry, pass feedback into the next prompt, e.g. "The previous attempt used
these banned phrases: X. Rewrite without them." To make this testable, change
the injected NarratorFn surface or wrap the real API call so it can receive
retry context. Alternatively, set an explicit temperature and document why
retries are expected to differ. Do not leave retry success dependent on provider
defaults.

5. Medium: CODEX_PROMPT.md's py_compile verification still fails under zsh.

Status: OPEN, review protocol command bug

Details:
review/CODEX_PROMPT.md replaced the doc-only unsafe py_compile command, but the
new block still assumes shell word splitting of a newline-separated scalar:
    PY_FILES=$(git diff --name-only "${CLOSING_COMMIT}..HEAD" | grep -E '\.py$' || true)
    python3 -m py_compile $PY_FILES

In this repo's zsh environment, unquoted scalar expansion does not split those
newlines into separate arguments. The command failed with:
    [Errno 2] No such file or directory: 'data/gen/agent_actor_mixer.py\ndata/gen/build_dataset.py\n...'

This is not a Batch 2 model/data bug, but it directly affected this review and
will affect future reviews that touch multiple Python files.

Recommendation:
Use an xargs or array-based command that is shell-robust:
    git diff --name-only "${CLOSING_COMMIT}..HEAD" \
      | grep -E '\.py$' \
      | xargs -r python3 -m py_compile
On macOS/BSD where `xargs -r` may not exist, use a temporary file or a Python
wrapper to read newline-separated paths and call py_compile.

6. Medium: INDEX.md still causes the official closing-commit extractor to
diff from the wrong base.

Status: OPEN, review protocol command bug

Details:
review/CODEX_PROMPT.md now extracts the last 7-hex token anywhere in
review/INDEX.md. review/INDEX.md lines 31-33 mention `b903809` in the "Latest
open question" prose after the table. As a result, the official extractor
returns b903809, not the latest closing commit 940f5bc. The prompt therefore
pulls review-protocol restructuring and older commits into the diff for this
Batch 2 review.

Recommendation:
Restrict extraction to the table's "Closing commit" column, or keep all prose
hashes above the table. A robust reviewer command would parse table rows only,
for example with awk selecting rows that start with `| 00` and printing the
last field, then extracting the hash from that field. Also update the prose
"Latest open question" to avoid bare 7-character hashes after the table, or put
them in followup files only.
