1. High: LLM narration receives and can leak semantic actor labels.

Status: FIXED.

Action:

a. data/gen/narrative_generator.py: replaced the prompt's "Actor type:
   {actor}." line with a neutral behavioral descriptor. New constant
   `_ACTOR_CADENCE_DESCRIPTORS` maps each of the 6 actor families to a
   class-name-free cadence string:

       human             -> "human-paced; irregular inter-event intervals"
       agent_buying      -> "programmatic; tool-mediated steps; moderately fast cadence"
       agent_finance     -> "programmatic; tool-mediated steps; regular cadence"
       agent_compromised -> "very fast; highly regular cadence; tool-mediated steps"
       agent_adversarial -> "fast; regular cadence with mimicry jitter; tool-mediated steps"
       hybrid            -> "mixed human-paced and tool-mediated steps"

   The narrator now sees "Interaction cadence: <descriptor>." instead of
   the raw class name. The cadence words alone do not encode
   class identity in a way the model can reliably decode at eval time.

b. data/gen/narrative_generator.py SYSTEM_PROMPT rule 3 rewritten:

       Refer to the actor as one of: "the account holder", "the actor",
       "the session". For agent-mediated sessions you may use
       "the agent" or "the tool-mediated agent". You must NEVER use any
       class label (compromised, adversarial, buying assistant, finance
       assistant, hybrid agent, etc.). Describe BEHAVIOR (cadence,
       tool-use trace), not CLASS.

   Rule 1's banned-stem list was extended in the prompt with the
   newly-banned actor-class words/phrases.

c. eval/leakage_checks.py `_BANNED_STEMS` extended:

       \bcompromised\b
       \badversarial\b
       \bmalicious\s+(agent|bot|tool|assistant)\b
       \b(buying|shopping)\s+(assistant|agent|bot|tool)\b
       \b(finance|financial)\s+(assistant|agent|bot|tool)\b
       \bhybrid\s+(actor|agent|user|session)\b
       \bagent_(buying|finance|compromised|adversarial)\b

d. data/gen/cheap_template_generator.py: the agent-cadence notes for
   `agent_buying` and `agent_finance` previously used the phrases
   "shopping assistant" and "financial assistant" — both now banned.
   Rewrote them to use neutral cadence descriptors. Verified all 54
   (9 family × 6 actor) combinations leakage-clean.

e. data/gen/cheap_template_generator.py self-test: now exhaustive (54

3. Medium: CostTracker can accept and cache a narrative after exceeding budget.

Status: FIXED.

Action: data/gen/narrative_generator.py `generate_narrative` now has a
post-`tracker.charge(...)` budget check inside the retry loop. If the
charge took spent_usd over budget_usd, it raises RuntimeError BEFORE
the fence/scan/cache_put steps. The in-flight call's cost is sunk
(unavoidable: the API call already happened), but the narrative is
rejected and not cached.

The minimum-fix wording from the reviewer's recommendation is the
implementation; we did not add a pre-call estimated-cost guard, which
is documented as a known limitation if precision is needed later.

Verify (the reviewer's reproducer):

    python3 -c "

    import tempfile
    from data.gen.journey_templates import generate
    from data.gen.narrative_generator import CostTracker, generate_narrative

        return ('Operational activity described without banned terms.', 1000000, 1000000)

    tracker = CostTracker(budget_usd=0.5)
    with tempfile.TemporaryDirectory() as td:
        try:
            generate_narrative(j, tracker=tracker, narrator_fn=costly,

    "

4. Medium: LLM retry loop is likely to repeat the same leaky output.

Status: FIXED.

Action:

a. data/gen/narrative_generator.py: `NarratorFn` signature changed to:

       Callable[[Journey, int, list], tuple[str, int, int]]

   where the new args are `attempt_num` (0-indexed) and `prev_hits` (the
   `narrative_leakage_scan` hits from the previous attempt; empty on
   initial call). This makes retry context explicit at the type level.

b. `_real_anthropic_call` now uses the retry context:
   - On `attempt == 0`: temperature=0.3, vanilla SYSTEM_PROMPT.
   - On `attempt > 0`: temperature=0.7 AND a retry-feedback addendum
     appended to the system prompt naming the banned phrases that were
     detected last time:

         RETRY FEEDBACK (attempt N): a previous attempt at this narrative
         was REJECTED because it used these banned phrases or stems:
         {phrases}. Rewrite the narrative without using any of them or
         their variants. Stay strictly within the HARD RULES above.

   Both temperature and prompt variation are needed: temperature alone
   sometimes still hits the same regional attractor; feedback alone with
   temperature=0 risks repeated stuck outputs.

c. `_stub_narrator` updated to accept (and ignore) the new args so
   self-tests still pass.

Verify (signature only — full LLM verify requires API key + a leaky
journey, which is exercised on the pod):

    python3 -c "

    print('stub OK; returns', type(out).__name__, 'of len', len(out))

Known follow-on: the stub narrator does not exercise retry-feedback
