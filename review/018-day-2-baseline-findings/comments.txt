Summary
=======

The three Day-2 findings are real. The current baseline leaderboard should not be used to decide Task #37 or to seed the auto-loop as-is.

Finding 1 is confirmed exactly: 533 of 5000 eval rows, 10.7%, have identical narrative text in train. The duplicates are concentrated in clean and hard-negative families, including 35.3% of hn_large_purchase and 16.4% of hn_account_recovery.

Finding 2 is confirmed, with one important correction. The current "largest achievable FPR <= 1%" rule measures event_only at 0.114% legit FPR while measuring the LM baselines near 1%. That is not a fair operating-point comparison. However, a naive percentile-threshold replacement using score >= threshold is also wrong when there are many tied scores; it moves event_only to 4.14% legit FPR and overstates the reversal. A tie-aware exact-1%-FPR metric still reverses the leaderboard, but less dramatically.

Finding 3 is confirmed. The current synthetic structured-event distribution is label-deterministic in the observed split. That is not a code bug, but it limits the claim the experiment can support unless documented and evaluated carefully.

Bottom line: fix this holistically before Task #37. Do not patch only leakage, and do not patch only the metric. The next valid step is: clean the eval view, replace the headline metric with a tie-aware low-FPR hard-negative metric, re-score existing baseline predictions, update sweep-state ranking to the corrected metric, and document the synthetic-data limitation.

Findings
========

1. CONFIRMS Day-2 finding: train/eval leakage invalidates the current eval rows

Severity: Blocker

The reported 10.7% text-hash leakage is correct. I reproduced 533 leaked eval rows out of 5000. The concentration by family is also correct: hn_large_purchase has 175 leaked rows out of 496, and hn_account_recovery has 80 out of 488.

This is a data-pipeline design issue, not a trainer bug. data/gen/build_dataset.py narrates the full generated set before the train/eval split, and data/gen/narrative_generator.py caches narratives by structured-event hash plus narrator settings. When the generator emits the same structured event sequence more than once, the cached narrative can be reused, and then the later stratified split can place identical narrative text on both sides.

The evidence is stronger than exact text duplication: every leaked row also has the same structured_events hash as a train row, and 4661 of 5000 eval rows share a bucket/event skeleton with a train row. The exact-text leak is the immediate Blocker. The skeleton overlap is a broader synthetic-distribution limitation to document and optionally diagnose later.

Recommendation: do not proceed with the current eval leaderboard. Build a clean eval view now by dropping rows whose narrative text hash or structured_events hash exists in train, then re-score existing predictions with the same row mask. For future generated datasets, fix this before narration by assigning disjoint structured examples to train/eval before LLM calls, or by enforcing structured-event disjointness before the cache can be reused across splits. A split-aware cache key alone would remove exact duplicated text but would not remove duplicated structured examples.

2. CONFIRMS Day-2 finding: current HN-FPR@1% implementation compares models at different operating points

Severity: Blocker

eval/score_risk.py currently uses sklearn's roc_curve and chooses the largest achievable FPR <= target. That is a faithful implementation of the old spec, but the old spec is incomplete for cliffed score distributions.

The current launcher-reported comparison is not apples-to-apples:

- event_only is measured at achieved legit FPR 0.001143, not 0.01.
- lora_text is measured at achieved legit FPR 0.009712.
- structured_as_text is measured at achieved legit FPR 0.009140.

That means event_only receives a much stricter legit threshold than the LM baselines. Its headline HN-FPR looks better partly because the operating point is different.

However, the proposed "top 1% of legit percentile threshold" needs a tighter definition before implementation. If implemented naively as score >= threshold, tied scores make event_only jump to achieved FPR 0.041417 and hn_account_recovery 0.2971. That is not an exact 1% operating point either; it is the same discretization problem in a different form.

The tie-adjusted exact-1% calculation gives a more defensible picture:

- event_only worst HN-FPR: 0.0717
- lora_text worst HN-FPR: 0.0556
- structured_as_text worst HN-FPR: 0.0497

After removing exact text leaks, the tie-adjusted comparison remains materially changed:

- event_only worst HN-FPR: 0.0730
- lora_text worst HN-FPR: 0.0701
- structured_as_text worst HN-FPR: 0.0507

Recommendation: replace the headline metric before any x-attn ranking. Use a tie-aware exact-target legit-FPR HN metric, or a deterministic fixed-rank metric with explicit tie handling. Record target FPR, effective FPR, threshold, number above threshold, number tied at threshold, and tie fraction. Also add a secondary low-FPR hard-negative curve or fixed-quantile table at 0.1%, 0.5%, and 1.0% legit FPR. Do not use plain fraud-vs-legit partial AUC as the only replacement; ordinary AUC is already saturated and does not directly answer the hard-negative false-positive question.

3. CONFIRMS Day-2 finding: synthetic structured data is label-deterministic in the observed split

Severity: High

This is not a code bug. It is a property of the synthetic distribution and feature design.

PLAN.md deliberately puts fraud signal into bucketed derived-feature tokens. That design is reasonable for a privacy-safe POC, and PLAN.md explicitly treats "event-only classifier wins" as a valid possible finding. But the observed dataset is stronger than merely "event-only friendly": in the generated train+eval data, journey_family alone has zero label entropy, and the bucket/event skeleton has zero conditional label entropy across 2454 observed skeletons.

That means the event-only classifier is not discovering a subtle behavioral signal. It is learning a compact categorical mapping over a structured stream where the label is deterministic in the observed support. The LM baselines read the same distribution through a harder interface, so direct event-only performance is not a clean proxy for "the LM is unnecessary in the real domain."

Recommendation: do not try to "fix" this during Day 2 by regenerating everything. Document it as a POC finding and constrain the Day-3 claim: results answer the question for this synthetic structured-feature distribution, not for real PayPal ATO data in general. If the project extends past this POC, add adversarial hard negatives with overlapping bucket patterns, mixed-label journey families, and a held-out skeleton split so event-only must generalize rather than memorize deterministic feature templates.

4. MISSES issue: the proposed percentile-threshold fix can introduce a second metric artifact unless ties are first-class

Severity: High

The pending brief correctly says the current sklearn threshold rule is unfair. It then proposes percentile-of-legit threshold as the cheap fix. That is directionally right, but incomplete.

The baseline artifacts show event_only has large tied score masses. At the percentile boundary, using score >= threshold includes all tied rows and overshoots 1% legit FPR by more than 4x. Using score > threshold undershoots. Either choice is arbitrary unless the metric defines how to allocate tied scores.

This must be fixed in the same pass as the headline metric. Otherwise the code may stop using sklearn's cliff and replace it with a different threshold cliff.

Recommendation: implement tie-aware expected counts, deterministic ranking with stable tie breaks, or report both lower and upper bounds for tied thresholds. My preferred path is tie-aware expected counts because it preserves the exact target FPR and can be bootstrapped.

5. MISSES issue: exact text dedup is necessary but not sufficient for future validity

Severity: Medium

Filtering the 533 exact narrative leaks is necessary for the current Day-2 leaderboard. It is not enough to claim robust generalization because 4661 of 5000 eval rows share the same bucket/event skeleton with a train row.

This does not need to block Task #37 if it is documented. The immediate comparison can proceed after exact text and structured-event duplicates are filtered. But the final writeup should clearly separate:

- clean-eval corrected leaderboard, which removes exact duplicate train/eval examples;
- synthetic-support limitation, where train and eval still share many feature templates;
- future extension work, where a harder split holds out skeletons or introduces mixed-label templates.

Recommendation: add a diagnostic count for text hash, structured_events hash, and bucket/event skeleton overlap. Fail on text/structured hash overlap for primary eval. Warn on skeleton overlap unless a future "held-out skeleton" eval set is explicitly created.

Holistic fix path
=================

Recommended order before Task #37:

1. Freeze the current leaderboard as stale.

Do not use the existing event_only, lora_text, and structured_as_text rows in src/auto_research/experiments.jsonl for ranking or auto-loop comparison without correction.

2. Create a clean eval mask/view.

Drop eval rows whose narrative text hash or structured_events hash exists in train. Apply the same row mask to existing baseline predictions and re-score. This is cheap because no retraining is required. The exact text leak removes 533 rows; structured-event hash removes 534 rows in this dataset.

3. Replace the primary metric.

Use tie-aware exact-target legit-FPR hard-negative FPR as the headline metric. The existing "largest achievable FPR <= target" rule is not fair across score distributions. A naive inclusive percentile threshold is also not fair with ties.

4. Update bootstrap and launcher state together.

The metric change has to land in eval/score_risk.py, eval/bootstrap_ci.py, scripts/run_next_experiment.py, and the sweep-state ranking fields. The auto-loop should optimize the corrected worst-family hard-negative FPR, not saturated AUC and not the old sklearn-threshold HN-FPR.

5. Re-score existing baselines before training x-attn.

Once the corrected metric and clean eval mask exist, re-score the existing predictions. Then decide Task #37 from the corrected table. The model weights do not need to change for this correction.

6. Document the synthetic-data property as an experimental result.

Do not hide the fact that the current structured distribution is label-deterministic. It is a real POC finding. The final claim should be about whether cross-attention beats simpler interfaces on this synthetic benchmark after leakage and metric artifacts are removed.

Bottom line
===========

The Day-2 findings are valid and material. The right fix is not an isolated data patch or an isolated metric patch. It is a coordinated correction: clean eval view, tie-aware low-FPR hard-negative metric, bootstrap/launcher/schema update, baseline re-score, and explicit documentation of the synthetic-data limitation.

Until that lands, Task #37 and the auto-loop would be optimizing against an invalid comparison surface.
