Review findings:

1. High: INDEX closing-commit format breaks the prompt's diff workflow.

Status: OPEN, review protocol ambiguity

Details:
review/CODEX_PROMPT.md lines 59-65 tell the reviewer to find the most recent
closing commit with:
    grep -E '\| [0-9a-f]{7}' review/INDEX.md | tail -1

review/INDEX.md line 12 stores review 002's closing commit as:
    67ee1e7 + b903809

That cell is human-readable but not a single commit value. A reviewer following
the prompt has to guess whether to diff from 67ee1e7, b903809, or parse both.
For this pass I used b903809 because it is the last hash and the index says
Batch 1 shipped there, but the prompt does not encode that rule.

Recommendation:
Make the "Closing commit" column contain exactly one hash: the final commit that
closes that review. If multiple commits matter, put the others in followup.txt,
or add a separate notes column. Also update CODEX_PROMPT.md to extract a single
hash mechanically, for example:
    CLOSING_COMMIT=$(grep -Eo '[0-9a-f]{7}' review/INDEX.md | tail -1)
    git log --oneline "${CLOSING_COMMIT}..HEAD"
    git diff --stat "${CLOSING_COMMIT}..HEAD"

3. Medium: CODEX_PROMPT.md's py_compile command fails on doc-only diffs.

Status: OPEN, verification command bug

Details:
review/CODEX_PROMPT.md lines 119-120 give:
    python3 -m py_compile $(git diff --name-only <closing-commit>..HEAD | grep -E '\.py$')

This diff changed only review docs. The grep returned no paths, so the command
expanded to python3 -m py_compile with no filenames and failed with exit code 2.
That creates a false negative for review-protocol or doc-only reviews.

Recommendation:
Use an empty-safe shell block:
    PY_FILES=$(git diff --name-only "${CLOSING_COMMIT}..HEAD" | grep -E '\.py$' || true)
    if [ -n "$PY_FILES" ]; then
      python3 -m py_compile $PY_FILES
    else
      echo "no Python files changed"
    fi

4. Minor: CODEX_PROMPT.md hard rule overstates the no-write policy.

Status: OPEN, doc wording only

Details:
review/CODEX_PROMPT.md lines 180-184 says the reviewer must not edit any file
other than comments.txt, then says not to run git add/commit/push "or any other
write operation against the git tree." Read literally, "any other write
operation" conflicts with Step 2's mkdir -p review/<NNN-slug> and Step 6's
comments.txt creation, both of which are required write operations inside the
worktree.

Recommendation:
Keep the intent but make the exception explicit:
    Do not run git add, git commit, git push, or modify any tracked project
    file. The only allowed worktree writes are mkdir for the new review folder
    and writing review/<NNN-slug>/comments.txt.

Bottom line:
The new review protocol is usable, but it should be tightened before the next
review pass. Fix the closing-commit extraction first; without that, every
future "diff since last closed review" is vulnerable to reviewer interpretation.
