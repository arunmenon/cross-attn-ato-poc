1. BLOCKER: launcher fail-open if clean-eval paths don't resolve

Status: FIXED.

Codex's concern: a future config with missing or unresolvable
data.train_path / data.eval_fast_path would cause run_post_processing
to silently fall back to scoring raw/leaky eval, while STILL emitting
a metric_version=2 row. validate_config doesn't reject configs with
unresolvable paths, so the failure mode is detection-only at the
review-eyes-on level — exactly the kind of silent reintroduction of
the review-018 leakage we corrected.

Action: scripts/run_next_experiment.py:706-728 (in run_post_processing):
The clean-eval-mask resolution check now runs FIRST (before
parse_metrics.py, before score_risk, before bootstrap_ci). If `cfg` is
provided (a real launcher run, not a stand-alone debug invocation) AND
the mask bundle can't be resolved, raise RuntimeError with the
specific config paths logged. This converts a silent-leakage fail-open
into a hard-stop fail-closed.

Specifically:
- If `cfg is None` (legacy debug path): clean_active=False, no mask,
  raw-eval fallback proceeds. Won't append rows.
- If `cfg is not None` AND `compute_and_cache_clean_eval_mask()`
  returns None (paths don't resolve): RAISE RuntimeError. The trainer
  output is preserved on disk but no metric_version=2 row is appended.

Verify (local):

    python3 - <<'PY'

    from scripts.run_next_experiment import run_post_processing
    from pathlib import Path
    import tempfile

        rd = Path(td) / "exp_test"; rd.mkdir()
        cfg = {"arm": "xattn", "data": {"train_path": "/nope", "eval_fast_path": "/nope"}}
        try:
            run_post_processing(rd, ["stripped"], cfg=cfg)
            print("FAIL: expected RuntimeError")
        except RuntimeError as e:
            assert "review 020 Blocker" in str(e)
            print("PASS: clean-eval fail-closed works")
    PY

Verified post-fix on the pod: dry-run-validate-clean-eval against the
real baseline config still resolves correctly (clean_eval_n=4466,
clean_eval_dropped=534) — the happy path is unchanged.

Status: FIXED.

Codex's concern: the comment at data/gen/build_dataset.py:485-489
said the split decision is made BEFORE narration, but the actual
order is narrate → assign_disjoint_splits → assemble records. Not a
correctness bug (the disjointness invariant depends on atomic group
assignment by events_hash, not on temporal ordering of narration vs
split), but a documentation/code mismatch that would mislead future
maintainers.

Action: data/gen/build_dataset.py:485-499. Rewrote the comment block
to:
- Drop the "Pre-narration" misnomer (the split is post-narration but
  pre-record-assembly).
- Explain that the disjointness invariant comes from atomic group
  assignment, not from narration ordering.
- Add an explicit "for future regenerations" note that moving the
  split BEFORE narration would avoid paying narration cost on rows
  that end up in the opposite split — an optimization, not a
  correctness fix.

No code change; comment-only.
