Findings

1. BLOCKER: New launcher rows can still be marked metric_version=2 after scoring the raw, leaky eval if the clean-eval mask cannot be resolved.

Files:
scripts/run_next_experiment.py:117-145
scripts/run_next_experiment.py:597-663
scripts/run_next_experiment.py:682-697

The launcher applies the clean-eval mask only if compute_and_cache_clean_eval_mask() returns a bundle. If paths are missing or unresolvable, _resolve_clean_eval_paths() returns None, run_post_processing() falls back to raw predictions, and the record builder still writes metric_version: 2. validate_config() does not require data.train_path or data.eval_fast_path, and it also does not reject unresolvable paths.

Verification:
python3 -B - <<'PY'

print(validate_config({'arm': 'xattn', 'data': {'train_path': '/missing/train', 'eval_fast_path': '/missing/eval'}}))
PY

Output:
[]
[]

Why this matters:
Review 019 Blocker 2 was specifically about ensuring every new run inherits the corrected clean-eval surface. This implementation handles the happy path, and the pod dry-run proves the current baseline config resolves correctly, but it is fail-open. An agent-written Task #37/#38 config with a missing or misspelled path could train, score on leaked eval, append metric_version=2, and enter sweep_state as if it were comparable to the corrected rows.

Required adjustment:
For metric_version=2 rows, missing/unresolvable clean-eval paths should be a hard launch or post-processing failure, not a raw-eval fallback. Alternatively, validate_config() must require resolvable data.train_path plus eval_fast_path/eval_path for all arms that emit v2 metrics.

File:

This is not reopening the leakage bug by itself: because the split groups identical structured_events hashes atomically, cross-split structured-events leakage should still be prevented, and Gate B asserts no cross-split text overlap after records are assembled. But the code/comment mismatch matters for future data-regeneration expectations and for AC6 wording. The implementation prevents cross-split leakage; it does not avoid narration/cache work before the split.

Answers to the 8 changelog questions
1. Tie-aware formula correctness: Yes. The implementation uses fractional need = target_fpr * n_legit and alpha = (need - n_above) / n_tied, clamped to [0, 1]. Pod selftests cover the review-019 event_only-style fixture.

2. Bootstrap stochastics: Production bootstrap defaults seed=0, so rescores are deterministic unless a caller overrides seed. The selftest and CI bundles are stable.

3. Halt-filter extension scope: Yes, max_experiments is still affected. See Finding 2.

4. Status semantics for rescored rows: Yes, the script should emit status="ok" plus provenance="rescored_v2". See Finding 3.

5. Intra-split diagnostic vs hard assertion: The diagnostic-only choice is acceptable for current scope. Review 018 identified cross-split leakage as the validity blocker; intra-split duplicates are expected under the saturated synthetic distribution and should be documented, not hard-failed.

6. Disjoint stratification class balance: Best-effort group assignment is acceptable for the current guardrail, as long as actual_eval_frac is surfaced. For future data regeneration, low-diversity families may need a tighter balancing algorithm, but I would not block this fix pass on it.

7. AC12 mechanical verification: The remote find check is sufficient for "no new run directories." An audit log of artifact files would be nice, not required.

8. Future Task #37 readiness: Not fully. The current standard pod config resolves the clean-eval mask correctly, and current sweep_state will rank v2 rows correctly. But the launcher is not fail-closed, so a fresh x-attn config can still produce a metric_version=2 row on raw/leaky eval if paths are missing or unresolvable.
