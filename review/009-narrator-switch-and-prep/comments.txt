Review findings:

1. High: LLM_PROVIDER=anthropic dispatches to Anthropic with the OpenAI default model.

Status: OPEN, should be fixed before relying on the documented provider fallback.

Details:
`data/gen/narrative_generator.py` documents two provider choices:
`LLM_PROVIDER=openai` with `OPENAI_API_KEY` maps to gpt-5.4-nano, and
`LLM_PROVIDER=anthropic` with `ANTHROPIC_API_KEY` maps to
claude-haiku-4-5 (`data/gen/narrative_generator.py:27-30`). The constants
also define `DEFAULT_ANTHROPIC_MODEL = "claude-haiku-4-5-20251001"`
(`data/gen/narrative_generator.py:69-71`). The actual dispatch does not use
that default. `generate_narrative()` defaults `model` to `DEFAULT_MODEL`, which
is always `DEFAULT_OPENAI_MODEL` (`data/gen/narrative_generator.py:87-89` and
`data/gen/narrative_generator.py:426`). When `LLM_PROVIDER=anthropic` is set,
the code switches only the client factory, not the model string
(`data/gen/narrative_generator.py:466-471`).

The provider-dispatch probe monkeypatched the Anthropic factory to avoid a real
network call and printed the model it would receive. It printed
`['gpt-5.4-nano']`. In a real run, Anthropic will be asked to serve an OpenAI
model name. `build_dataset.py` also exposes no `--model` flag, so users cannot
repair this from the documented CLI (`data/gen/build_dataset.py:346-355`).

Recommendation:
Make provider selection and model selection a single coherent contract. The
simplest fix is: if `model` was not explicitly supplied and
`LLM_PROVIDER=anthropic`, set `model = DEFAULT_ANTHROPIC_MODEL`; if
`LLM_PROVIDER=openai`, set `model = DEFAULT_OPENAI_MODEL`. Also add a
`--llm-model` flag to `build_dataset.py` or explicitly document that Anthropic
fallback requires a Python-level call with `model=DEFAULT_ANTHROPIC_MODEL`.
Add a no-network unit probe that asserts the Anthropic factory receives
`claude-haiku-4-5-20251001` under `LLM_PROVIDER=anthropic`.

2. Medium: RUNBOOK.md gives the wrong training filename for eval-frac builds.

Status: OPEN, doc/code drift in the pod data-prep instructions.

Details:
RUNBOOK.md says the 25k LLM command with `--eval-frac 0.2` writes
`train_llm_narrated/data.jsonl` plus `train_llm_narrated/eval.jsonl`
(`RUNBOOK.md:128-134`). That is not what `build_dataset.py` does. When
`eval_frac > 0`, it writes `train.jsonl` and `eval.jsonl`
(`data/gen/build_dataset.py:285-297`). The verification run confirms the actual
file list: `train.jsonl`, `eval.jsonl`, and `build_summary.json`, with no
`data.jsonl`.

The code-side loader is correct: `src/train/common.py` explicitly treats
`train.jsonl` plus `eval.jsonl` as the split-output layout and `data.jsonl` as
the single-file layout (`src/train/common.py:78-113`). This is a runbook defect,
not a dataset-builder defect. It still matters because the pre-pod upload and
sanity-check workflow is supposed to be copy-pasteable before moving data to
RunPod.

Recommendation:
Change the data-tree diagram and §0.3 prose to say
`train_llm_narrated/train.jsonl + train_llm_narrated/eval.jsonl` when
`--eval-frac` is nonzero. Keep the existing `eval_fast_5k` symlink target
pointing at `../train_llm_narrated/eval.jsonl`; that part is consistent with
the code.

3. High: The tokenizer/preflight section claims checks that preflight_check.py does not perform.

Status: OPEN, should be fixed before treating pod preflight as the Qwen3 tokenizer gate.

Details:
RUNBOOK.md says Part B of tokenizer sanity runs `scripts/preflight_check.py`
for "AutoTokenizer roundtrip + resize" and then states that it exercises
`AutoTokenizer.from_pretrained(...)`, embedding-average initialization, and
`model.resize_token_embeddings(...)` (`RUNBOOK.md:112-116`). The script does
not load a model, does not call `src.tokenizer.custom_tokens.install()`, does
not install all 65 custom tokens, does not initialize new rows, and does not
call `model.resize_token_embeddings(...)`. Its tokenizer check adds only six
literal tokens to the tokenizer object and checks decode roundtrip
(`scripts/preflight_check.py:88-107`).

This also misses the label-token contract that review 007 made load-bearing.
`src/train/common.get_label_token_ids()` fails fast if " fraud" or " legit" is
empty, multi-token, or the same ID (`src/train/common.py:122-164`), but
preflight never calls that helper. A pod can pass preflight and still fail only
when the first trainer/eval path reaches label scoring.

Recommendation:
Either narrow the RUNBOOK wording to what preflight actually checks, or upgrade
`scripts/preflight_check.py` into the gate the runbook describes. For pod
readiness, the better fix is to add a tokenizer/model-resize check that:
loads the Qwen3 tokenizer, calls `src.tokenizer.custom_tokens.install(tok)`,
asserts all 65 custom tokens are present without collisions, calls
`get_label_token_ids(tok)`, loads the base model when GPU memory is available,
calls `model.resize_token_embeddings(len(tok))`, and verifies the embedding
row count equals `len(tok)`. If the full model load is intentionally deferred,
split this into `check_tokenizer_contract()` and `check_model_resize_contract()`
so the runbook stops implying both happen in today’s preflight.

4. Medium: The "pure-Python local smoke" asks a laptop to install the full pod requirements.

Status: OPEN, local-prep ergonomics/documentation issue.

Details:
RUNBOOK.md labels §0.1 "Pure-Python scaffold smoke (no torch needed)" and says
`pip install -r requirements.txt` "installs openai + light deps"
(`RUNBOOK.md:93-103`). `requirements.txt` is not a light local-prep file. It
pins the full HF/training stack: `transformers`, `accelerate`, `peft`,
`datasets`, `tokenizers`, `safetensors`, and `bitsandbytes`
(`requirements.txt:4-13`), plus W&B and both narrator SDKs
(`requirements.txt:25-34`). On this review host none of `openai`, `anthropic`,
`transformers`, or `bitsandbytes` were installed before the smoke; the actual
torch-free self-tests passed without installing the pod stack because they did
not need those imports.

This is not a model-training bug, but it weakens the pre-pod workflow. A user
trying to do only the cheap local prep on macOS can get dragged into installing
GPU-oriented packages before any API or template-data generation step. That is
the opposite of the "H100 clock starts on training, not setup" goal in
RUNBOOK.md:91.

Recommendation:
Split dependencies by use. Add a small local data-prep requirements file
covering the narrator SDK(s), `faker`, `tqdm`, and the pure-Python package
needs, or change §0.1 to install only the specific package(s) needed for the
chosen local step. Keep `requirements.txt` as the pod/training requirements if
that is the intended contract. Also remove or soften the claim that the
narrative self-test verifies the full pricing-table math; it only exercises the
stub path and prints the current charge.

Bottom line:
Do not block the OpenAI default on model availability; that part checks out.
Fix the provider fallback and the misleading runbook steps before handing this
to RunPod as a copy-paste local-prep path.
