1. High: LLM_PROVIDER=anthropic dispatches to Anthropic with the OpenAI
   default model.

Status: FIXED.

Action: rewrote the resolution path in narrative_generator.py.

  - New helper `_default_model_for_provider(provider) -> str` maps
    "openai" -> DEFAULT_OPENAI_MODEL, "anthropic" -> DEFAULT_ANTHROPIC_MODEL,
    else raises with a clear message.

  - New helper `_resolve_provider_and_model(model, env_override)` is the
    single source of truth for picking (provider, model). Resolution
    rules:
        no model + no env  -> DEFAULT_PROVIDER + that provider's default
        env only           -> env's provider + that provider's default
        model only         -> infer provider from name prefix
        both               -> infer from model; if env disagrees, raise

  - `generate_narrative()` signature changed: `model: str = DEFAULT_MODEL`
    -> `model: str | None = None`. The function now calls
    `_resolve_provider_and_model(...)` at the top, then uses the
    resolved (provider, model) pair for BOTH the cache key and the
    client factory. They can no longer drift apart.

  - The disagreement raise uses ValueError, not RuntimeError, because
    it is a programmer/config error, not a runtime budget failure.

CLI surface (build_dataset.py):

  - Added `--llm-model` flag (e.g., gpt-5.4-nano, gpt-5.4-mini,
    claude-haiku-4-5-20251001). Passed through to `build(...)` which
    forwards to `llm_narrative(..., model=llm_model)`. Default None ->
    provider default.

  - Added `--llm-provider {openai,anthropic}` convenience flag. Sets
    os.environ["LLM_PROVIDER"] before calling build, equivalent to
    prepending `LLM_PROVIDER=...` to the command line.

  - Provider/model agreement is enforced inside generate_narrative()
    via the same _resolve_provider_and_model() path, so the CLI cannot
    bypass the check.

3. High: The tokenizer/preflight section claims checks that
   preflight_check.py does not perform.

Status: FIXED (doc narrowed to match code).

Action: rewrote §0.2 Part B paragraph to describe what preflight
actually does and to name the real tokenizer-contract gate (which lives
in the trainer).

  - Removed the misleading "AutoTokenizer.from_pretrained + resize +
    embedding init" claim. Replaced with "tokenizer download + 6-token
    decode roundtrip" -- the literal scope of
    scripts/preflight_check.py:check_tokenizer_roundtrip.
  - Added explicit "what Part B does NOT do" list: no
    custom_tokens.install(), no 65-token install, no model load, no
    resize_token_embeddings(). Marked review 009 finding #3.
  - Pointed at src/train/common.py:get_label_token_ids() as the
    fail-fast contract for " fraud"/" legit" tokenization (already in
    place from review 007 finding #1) and at the trainers' explicit
    model.resize_token_embeddings(len(tok)) after custom_tokens.install.
  - Explicit warning: "A pod that passes preflight can still fail at
    first trainer launch if the embedding-resize path regresses, so
    treat the trainer smoke (Task #32) as the real gate."

Reviewer's recommended upgrade -- extending preflight_check.py into the
full tokenizer-contract gate -- is DEFERRED. Doing so requires loading
the Qwen3 base for resize_token_embeddings() to be meaningful, which is
~16 GB of weight download and several minutes of wall time even before
the trainer starts. The trainer already calls those exact lines at
startup (src/train/train_xattn.py and friends), so escalating preflight
would duplicate the same checks, double the boot time, and only catch
problems that the trainer would also catch within 30 seconds of its
own launch. The doc narrowing is the surgical fix; the upgrade would be
a separate enhancement.

Not part of this followup but worth noting for future review:
The narrator's OpenAI path computes cost from `usage.prompt_tokens` and
does not pass `cached_input_tokens` through to `CostTracker.charge`
yet. The NarratorFn signature returns (body, in_toks, out_toks) only.
This was acknowledged in narrative_generator.py:346-352 as a deferred
refactor. The visible effect at 25k narratives is that we slightly
over-estimate spend when prompt-caching fires (~10-15% high), which is
conservative for budget purposes and does not affect any blocking
behavior. Not blocking review 009 close; flag for review 010 if it
ever matters.
