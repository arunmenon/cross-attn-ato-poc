Review findings:

1. Blocker: CONFIRMS audit-014 finding #1 - crashed trainer can be recorded as `status="ok"`.

Status: CONFIRMS audit-014 finding #1, must-land

Details:
The audit severity is correct. This directly poisons unattended operation because status is the first gate used by sweep_state recomputation, top-3 selection, and halt accounting. The local reproduction above confirms the exact failure path: the launcher has a failed subprocess path, but parse_metrics can report `status: ok`, and the launcher currently trusts that parsed status when a metrics file exists.

Fix-correctness verdict:
The audit recommendation is directionally correct: the subprocess return code must remain authoritative unless the launcher positively proves a successful trainer completion. Tighten the implementation around that principle; do not let parse_metrics downgrade a launcher-level failure back to ok.

Recommendation:
Keep finding #1 in the must-land list before unattended cron activation.

2. Blocker: CONFIRMS audit-014 finding #2 - stale GPU lock wedges the loop after SIGKILL or pod preemption.

Status: CONFIRMS audit-014 finding #2, must-land

Details:
The audit severity is correct. `run_next_experiment.py` writes a PID-stamped lock with O_EXCL, and `agent_tick.sh` now politely exits before invoking the CLI when the file exists. That pre-check saves quota during normal runs, but it makes stale locks more visible: after an unclean launcher death, every future tick sees only the file and exits. Neither code path consults the PID with `kill -0`, so the lock cannot self-heal.

Fix-correctness verdict:
The recommended stale-PID cleanup is the right scope. It should live in the authoritative lock path, not only in the shell pre-check, because the launcher is the concurrency gate.

Recommendation:
Keep finding #2 in the must-land list before unattended cron activation.

3. Blocker: CONFIRMS audit-014 finding #3 - the agent has no enforced filesystem containment.

Status: CONFIRMS audit-014 finding #3, must-land

Details:
The audit severity is correct for unattended cron. AGENT_INSTRUCTIONS.md says the launcher owns experiments.jsonl and sweep_state.yaml, but that is convention, not enforcement. The tick invokes a general Claude CLI session on the pod. Without an explicit allowlist/denylist boundary, a confused agent can edit source, mutate state files, delete artifacts, or expose credentials outside the intended `runs/exp_NNN/config.yaml` and notes surface.

Fix-correctness verdict:
The audit recommendation is correct in shape: add an enforced repo-level tool/filesystem boundary before cron. One caveat: the exact `.claude/settings.json` schema and whether it applies to the pod's installed Claude CLI version should be validated on the pod, because a syntactically present settings file is not containment unless the CLI actually honors it.

Recommendation:
Keep finding #3 in the must-land list before unattended cron activation.

4. High: CONFIRMS audit-014 findings #5 and #6 - the halt window and process group kill path both need pre-cron fixes.

Status: CONFIRMS audit-014 findings #5 and #6, must-land

Details:
Finding #5 is real: if an otherwise status-ok row lacks `hn_fpr_worst_stripped`, the convergence branch silently skips halt for that window. The audit phrase "forever" is too broad if later windows stop containing None, but the bug is launch-relevant because finding #1 can create malformed status-ok rows. The local probe reproduced this.

Finding #6 is also real. The launcher starts `accelerate launch` without a new process session and kills only the direct child on timeout. In a DDP/dataloader tree, this can leak child workers that continue holding GPU memory after the launcher thinks the experiment is over. That is exactly the kind of failure that makes a cron-driven loop unreliable.

Fix-correctness verdict:
Both recommended fix directions solve the actual issue: make missing primary metrics explicit invalid/failure state for halt/ranking, and terminate the subprocess group rather than only the parent process.

Recommendation:
Keep findings #5 and #6 in the must-land list before unattended cron activation.

5. High: MISSES stale AUC objective in AGENT_INSTRUCTIONS.md.

Status: MISSES issue, add to same fix pass

Audit 014 did not flag that `src/auto_research/AGENT_INSTRUCTIONS.md` still tells the proposer to identify top-2 configs by AUC-stripped, documents convergence as "no AUC-stripped lift", and uses AUC-led result tables. 6. Medium: DISPUTES audit-014 finding #4 - checkpoint atomicity is worthwhile hardening, but not a High must-land as framed.

Status: DISPUTES audit-014 finding #4, severity over-calibrated

Details:
The underlying observation is true: `src/train/train_xattn.py` writes `xattn_state.pt` directly with `torch.save`, not temp-and-rename. Atomic checkpoint writes are a good practice.

The audit's race framing is too strong under the current backup ordering. `backup_to_external.sh` syncs top-3 run directories by reading `sweep_state.yaml.top_3`. A just-started or currently-training run is not in top_3 until after train_xattn.py returns, launcher post-processing runs, experiments.jsonl is appended, and sweep_state.yaml is recomputed. In normal flow, backup cannot select the current run's `xattn_state.pt` while it is being written because the run has not yet entered top_3.

This does not mean the recommendation is bad; temp-and-rename for a large checkpoint is cheap and sensible. It means this item should be downgraded to Medium/nice-to-have or reframed as defense-in-depth, not a High item that blocks cron activation.

Recommendation:
Remove finding #4 from the must-land list, or re-severity it to Medium. Keep the hardening if it is cheap, but do not count it as one of the launch blockers.
