3. Medium: narrator_temp does not participate in the cache key.

Status: FIXED (namespaced).

Action: extended _journey_cache_key() in data/gen/narrative_generator.py
to optionally include `narrator_temp` in the canonical dict.

Design choice: backward-compatible namespacing. When the caller's
narrator_temp equals the default 0.3, the cache key uses the OLD
format (no temp field) so existing ~30k cache entries from the 25k
train run still hit without regeneration. When narrator_temp != 0.3,
the temp value is added to the canonical dict, producing a fresh key
that won't collide with the default-temp cache. This is the
"namespace by temperature" option the reviewer called out, with the
constant DEFAULT_NARRATOR_TEMP_FOR_CACHE = 0.3 documenting the
backward-compat boundary.

Both call sites updated:
  - generate_narratives_concurrent._one (concurrent batch path,
    data/gen/narrative_generator.py:577)
  - generate_narrative (single-record path,
    data/gen/narrative_generator.py:684)

Verification (no-torch local probe):

    python3 -c "
    import sys; sys.path.insert(0, '.')
    from data.gen.narrative_generator import _journey_cache_key
    from data.gen.journey_templates import generate as gen_journey
    j = gen_journey('clean', seed=1, actor='human')
    k_default = _journey_cache_key(j, 'gpt-5.4-nano')
    k_03 = _journey_cache_key(j, 'gpt-5.4-nano', narrator_temp=0.3)
    k_05 = _journey_cache_key(j, 'gpt-5.4-nano', narrator_temp=0.5)
    assert k_default == k_03, 'default must equal temp=0.3 for back-compat'
    assert k_03 != k_05, 'different temps must produce different keys'
    print(f'default==temp_0.3=={k_03[:12]}..., temp_0.5={k_05[:12]}...')
    "

Operational note about the salvaged 15k eval (eval_medium_15k_llm):
The pre-fix gen run was launched with `--narrator-temp 0.5` but
served 100% cache hits from the previous train-time temp-0.3 cache.
So the salvaged 15k contains TEMP=0.3 narratives, not temp=0.5. The
diagnostic in flight against this eval will tell us what saturation
looks like on a seed-different but narrator-style-identical eval set.
Documented in experiments.jsonl entry for exp_stage0_001 in the next
review followup.

If a future eval gen wants ACTUAL temperature-0.5 narratives, the
fix landed here makes it work (key namespacing forces fresh API
calls, and the rate-limit-friendly concurrency=4 + lower n is the
recipe).

4. Medium: preflight_xattn proceeds on CPU instead of failing closed
   when CUDA is unavailable.

Status: FIXED.

Action: added an explicit CUDA gate at the top of
scripts/preflight_xattn.py:main() right after the torch import, before
loading the merged checkpoint:

    if not torch.cuda.is_available():
        print("FAIL: CUDA not available — preflight_xattn requires "
              "a GPU. Re-run scripts/preflight_check.py first to "
              "diagnose.", file=sys.stderr)
        return 4

Exit code 4 is reserved for "CUDA-substrate missing" so a caller can
distinguish it from forward failure (exit 2) and NaN loss (exit 3).

Also added substrate logging immediately after the gate:

    print(f"[xa-pre] torch.version.cuda={torch.version.cuda}, "
          f"device={torch.cuda.get_device_name(0)}, "
          f"compute_capability={torch.cuda.get_device_capability(0)}")

That captures the same context the bnb sanity heredoc in RUNBOOK §10
emits, so a future failure log diff between the two reveals substrate
mismatches at a glance.

The device assignment line later in the script is now a hardcoded
`device = "cuda"` with an explanatory comment pointing back to the
gate. No silent CPU fallback remains.
