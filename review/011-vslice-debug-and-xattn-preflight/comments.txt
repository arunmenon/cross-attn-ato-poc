Review findings:

1. Blocker: Event-consuming trainers never parse structured_events after load_paired_dataset turns it into JSON text.

Status: OPEN, blocks Task #36/#37 x-attn and event-side baselines.

Details:
`load_paired_dataset()` now serializes `structured_events` lists to JSON
strings before building the HF dataset (`src/train/common.py:145-160`). This
solves the pyarrow heterogeneous-list failure, but it changes the row shape.
`src/train/common.py` adds `parse_structured_events(row)` and its comments say
event-consuming arms should call it (`src/train/common.py:193-210`).

The callers do not. The only `rg parse_structured_events` hits are the comment
and the function definition. `train_xattn.py` still passes
`ex.get("structured_events", [])` directly into `tokenize_events`
(`src/train/train_xattn.py:43-45`). `train_event_only_classifier.py` does the
same with `ex["structured_events"]` (`src/train/train_event_only_classifier.py:50-52`).
`train_structured_as_text.py` slices the JSON string as though it were a list
of event dicts and then passes it to `event_to_line` (`src/train/train_structured_as_text.py:80-82`).

The command probe shows the concrete failure mode: `tokenize_events()` on the
new JSON-string shape raises `AttributeError: 'str' object has no attribute
'get'`. Parsing with `parse_structured_events(row)` first produces valid event
IDs. The CPT-only and LoRA-text arms can still run because they do not consume
events, which matches the reported live vertical-slice success; the next
event-side arms will fail.

Recommendation:
Wire `parse_structured_events()` into every event-consuming path:
`train_xattn._build_structured_collate`, `train_structured_as_text`'s collator
and eval pre-transform loop, `train_event_only_classifier.collate_classifier_batch`,
and any `structured_input_fn` used by `eval_runner`. Add a no-torch unit probe
that constructs a row with JSON-string `structured_events` and verifies
`tokenize_events(parse_structured_events(row), EventVocab())` succeeds. Add a
torch-side collator smoke on the pod if possible.

2. High: RUNBOOK still tells CUDA-13 Blackwell users to reinstall the old bitsandbytes <0.46 range.

Status: OPEN, pod recovery instructions contradict the new dependency fix.

Details:
Commit 4bf30f3 updates `requirements.txt` to `bitsandbytes>=0.45.0,<0.50`
because the RunPod Blackwell image has a CUDA 13 runtime and the old 0.45.x
range falls back to CPU-only on that image (`requirements.txt:20-30`). RUNBOOK
did not receive the same update. It still says `requirements.txt` pins
`>=0.45.0,<0.46` (`RUNBOOK.md:336`) and its first recovery command is:

    pip install --upgrade "bitsandbytes>=0.45.0,<0.46"

(`RUNBOOK.md:365`). That command is exactly the pre-4bf range the requirements
comment says lacks `libbitsandbytes_cuda130.so`. If the Blackwell bnb sanity
check fails and the operator follows the runbook, they can reinstall the broken
family and stay on the same failure path.

Recommendation:
Update RUNBOOK §10 to match `requirements.txt`: `bitsandbytes>=0.45.0,<0.50`,
and change the explanation from "may have selected a wheel without SM_120" to
the actual current target: CUDA-13-capable bitsandbytes, with 0.49+ expected on
the RunPod CUDA 13 image. The requirements line, runbook gotcha, and recovery
command should all be textually consistent.

3. Medium: narrator_temp does not participate in the cache key.

Status: OPEN, should be fixed before relying on temperature to make eval narration different from train narration.

Details:
The new CLI flag says `--narrator-temp` can be bumped to about 0.5 when
generating eval data to reduce narrator-style correlation with train
(`data/gen/build_dataset.py:406-414`). The factories now receive
`base_temperature`, so new API calls will honor it. The cache does not.
`_journey_cache_key()` hashes only model, journey family, actor family, and
events (`data/gen/narrative_generator.py:249-260`). Both sequential and
concurrent paths look up the cache before using `narrator_temp`
(`data/gen/narrative_generator.py:561-564` and `data/gen/narrative_generator.py:669-673`).

The verification probe generated the same journey once at temp 0.3, then
requested it again at temp 0.5 with a different narrator function. The second
call made zero narrator calls and returned the cached 0.3 text. That is fine
for cost-saving reruns, but it makes the new "hotter eval narration" control a
silent no-op whenever the structured journey already exists in
`data/cache/narratives/`.

Recommendation:
Either include `narrator_temp` in `_journey_cache_key()` or add a deliberate
cache-bypass/cache-namespace option for eval generation. If you intentionally
want cache reuse across temperatures, document that `--narrator-temp` only
affects cache misses and that operators must clear or namespace
`data/cache/narratives/` when regenerating the same structured journeys at a
new temperature.

4. Medium: preflight_xattn proceeds on CPU instead of failing closed when CUDA is unavailable.

Status: OPEN, operational guardrail gap in the new Stage-1 smoke.

Details:
`scripts/preflight_xattn.py` describes a GPU architecture smoke and says it
runs one batch "on GPU" in the header. The script chooses
`device = "cuda" if torch.cuda.is_available() else "cpu"` and continues
(`scripts/preflight_xattn.py:83-89`). On a broken pod where CUDA is invisible,
this will try to move a Qwen3 8B wrapper to CPU and run a 512-token forward,
which is not a useful preflight. The general `scripts/preflight_check.py`
should catch CUDA first, but this new script is explicitly a preflight and
should fail closed on its own.

Recommendation:
Add an explicit `if not torch.cuda.is_available(): FAIL/return nonzero` before
loading or moving the full model, and print a clear message directing the
operator back to `scripts/preflight_check.py`. Also consider printing
`torch.version.cuda`, device name, and capability here so the x-attn smoke
captures the same substrate context as the bnb smoke.

5. Minor: preflight_xattn exists but the runbook never tells the operator to run it.

Status: OPEN, doc wiring issue.

Details:
The new script is clearly intended to be the Task #35/Stage-1 architecture
gate. Its header says it catches wrapper, merged-checkpoint, event-tensor, and
LoRA-on-Q issues before spending Stage-1 GPU time. RUNBOOK Day 2 still says
only "X-attn architecture surgery (Task #35)" and then "First x-attn smoke run
(Task #37)" (`RUNBOOK.md:192-197`). There is no command for:

    python3 scripts/preflight_xattn.py --merged-checkpoint /workspace/checkpoints/qwen3-8b-cpt-light-merged

or guidance on where it sits relative to merge, baselines, and the first
x-attn trainer launch. That makes the new safety check easy to skip under the
time pressure it was created for.

Recommendation:
Add the exact `preflight_xattn.py` command to RUNBOOK Day 2 immediately after
the Stage-0 merge / CPT eval and before launching any Stage-1 x-attn training.
State the expected success line and the exit-code behavior, and point failures
at the wrapper/checkpoint/debug loop rather than the auto-research loop.

Bottom line:
Fix finding 1 before Task #36/#37; it is the event-side runtime break. Then
align the bitsandbytes recovery command with the CUDA-13 requirement and make
the new x-attn preflight fail closed and visible in the runbook.
