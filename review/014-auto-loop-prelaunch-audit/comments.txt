1. Blocker: Crashed trainer is recorded as `status="ok"` whenever train.log lacks an explicit NaN/OOM marker.

Status: OPEN, blocker

Details:
`scripts/parse_metrics.py:76` initializes `status = "ok"` and only
escalates to `"oom"` or `"nan"` when text-marker regexes match
(`OutOfMemoryError`, `CUDA out of memory`, `OOM`, `loss is nan`,
`gradient overflow`). For any crash that does not leave one of these
markers in train.log (host OOM-killer SIGKILL, accelerate launch error,
import error, segfault, env-var failure, SIGTERM, distributed deadlock,
bitsandbytes Blackwell incompat, etc.), `parse_log` returns
`status="ok"`. On the launcher side, `scripts/run_next_experiment.py:706-709`
does:

    metrics_json = run_dir / "metrics.json"
    if metrics_json.exists():
        m = json.loads(metrics_json.read_text())
        if not timed_out:
            record["status"] = m.get("status", "failed")

`m["status"]` is "ok" for any silent crash, so `record["status"]` — which
was correctly set to `"failed"` on line 698 based on rc != 0 — is
overwritten with `"ok"`. End-to-end repro confirmed (verification 5).
The bad row then:
  - Counts toward `xattn_valid_all` in `check_halt_conditions` (line 302),
    inflating `max_experiments` and `min_valid_runs_before_halt` counts.
  - Breaks the `nan_cascade` halt — a single mismarked "ok" mid-cascade
    silently disables the cascade detector, and the loop keeps burning
    GPU on NaN-producing configs.
  - Is invisible to the agent reading sweep_state.yaml.
  - Is wrong for any post-hoc audit / Day-3 writeup.

The audit brief explicitly called this out as a known-suspected bug
surfaced by Task #35 debugging. Independently verified by both the
correctness and state specialists.

Recommendation:
The fix belongs in `scripts/run_next_experiment.py:706-709`, not in
parse_metrics.py. parse_metrics's "ok"-by-default is reasonable on its
own — the launcher's failure branch trusting that default in a context
where rc != 0 is the bug. Three-line change:

    # rc != 0 here. Only let parsed status escalate to a more-specific
    # failure type (nan, oom). Never downgrade "failed" to "ok".
    if not timed_out:
        parsed = m.get("status")
        if parsed in ("nan", "oom"):
            record["status"] = parsed
        # else: keep "failed"

Defensive belt-and-suspenders in parse_metrics.parse_log:76 — change
the initial `status = "ok"` to `"unknown"` and escalate to "ok" only
when `train_runtime` is present in the log (HF Trainer emits this in
its summary line — proves the trainer reached the end). This is a
separate fix in parse_metrics; the launcher-side fix above is the
required one.

Verify:
    Create a synthetic crash scenario: trainer stub prints "step 1
    loss 0.5" then exits 1. Run launcher. Expect experiments.jsonl
    row has status=="failed", not "ok".

2. Blocker: Lockfile cannot self-heal across launcher SIGKILL / pod preemption.

Status: OPEN, blocker

Details:
`scripts/run_next_experiment.py:367-386` writes the launcher's PID into
`/workspace/.gpu.lock` using O_EXCL, and `release_lock` is wired into a
try/finally so a Python-level exception cleans up. But if the launcher
process itself is killed (OOM-killer SIGKILL, RunPod Community Cloud
preemption mid-experiment, ssh session kill, host blip) the finally
never runs and the lockfile remains. `scripts/agent_tick.sh:132-136`
pre-checks the file's existence — it does NOT check whether the PID
inside is still alive — so every subsequent 30-min tick reads "lock
held" and exits silently. The auto-loop is wedged until a human runs
`rm /workspace/.gpu.lock`.

For a 12-18 hour unattended run on Community Cloud (where RUNBOOK §10
documents preemption as "expected, not exceptional"), the expected
behavior of "the loop picks back up" only holds if the lock was
released cleanly — which is the case the host controls least. The
lockfile lives on `/workspace` (persistent RunPod volume), so even a
preemption-then-rebuild leaves the orphaned lock behind.

AGENT_INSTRUCTIONS.md line 199 tells the agent to "surface this to the
user via README — do not delete the lockfile yourself" — but the agent
is never invoked because agent_tick.sh's pre-check exits before the
CLI runs. Discovery-by-SSH-tail is a 30-min-to-12-hour outage depending
on operator schedule.

The three specialists framed this differently — security-auditor as
Blocker for unattended cron, state-auditor and best-practices as High —
but the highest-severity framing (Blocker) is the correct one for the
"is this safe to flip on for 12-18 hours unattended" question this audit
is scoping. Cited adoption in reference frameworks: Optuna
`fail_stale_trials`, Ray Tune cluster auto-restart, Anthropic
`kill-switch.sh` + `commit-on-stop`, OpenHands sandbox teardown — 4 of 7.

Recommendation:
Two-layer fix (both layers are short — belt-and-suspenders). In
`agent_tick.sh:132-136`, do a `kill -0 $LOCK_PID 2>/dev/null` test
gated on a >=120s lockfile-age threshold to avoid self-eviction races:

    if [[ -f "$GPU_LOCK_FILE" ]]; then
        LOCK_PID="$(cat "$GPU_LOCK_FILE" 2>/dev/null || echo unknown)"
        if [[ "$LOCK_PID" =~ ^[0-9]+$ ]] && ! kill -0 "$LOCK_PID" 2>/dev/null; then
            MTIME=$(stat -c %Y "$GPU_LOCK_FILE" 2>/dev/null || \
                    stat -f %m "$GPU_LOCK_FILE" 2>/dev/null || echo 0)
            NOW=$(date +%s)
            if (( NOW - MTIME > 120 )); then
                echo "[agent_tick] $(date -u +%Y-%m-%dT%H:%M:%SZ) reclaiming stale GPU lock (PID=$LOCK_PID is dead, age $(( NOW - MTIME ))s)"
                rm -f "$GPU_LOCK_FILE"
            else
                echo "[agent_tick] $(date -u +%Y-%m-%dT%H:%M:%SZ) lock PID=$LOCK_PID not alive but lock is recent ($(( NOW - MTIME ))s); waiting"
                exit 0
            fi
        else
            echo "[agent_tick] $(date -u +%Y-%m-%dT%H:%M:%SZ) GPU lock held (PID=$LOCK_PID); skipping this tick"
            exit 0
        fi
    fi

In `run_next_experiment.acquire_lock`, mirror the same `kill -0` check
on the FileExistsError branch, gated by the same 120s age threshold,
then atomically reclaim (unlink + retry once, not in a loop). Either
layer alone is sufficient; both is the dual-gate idiom the codebase
already uses elsewhere.

Verify:
    echo 99999 > /workspace/.gpu.lock           # fake PID
    touch -d '5 minutes ago' /workspace/.gpu.lock
    bash scripts/agent_tick.sh
    # Expect: "stale lock ... removing"; the tick proceeds.

Specialists contributing: security-auditor, state-auditor, best-practices-researcher (three-way corroborated).

3. Blocker: Agent has no enforced filesystem containment — `.claude/settings.json` does not exist.

Status: OPEN, blocker

Details:
CLAUDE.md / AGENT_INSTRUCTIONS.md / RUNBOOK.md collectively tell the
agent "do not edit run_next_experiment.py" (AGENT_INSTRUCTIONS:208), "do
not touch experiments.jsonl or sweep_state.yaml" (AGENT_INSTRUCTIONS:188),

all written conventions. Nothing in the running environment enforces them:

- No `.claude/settings.json` exists in the repo (`ls .claude/` shows

- agent_tick.sh runs `claude` as root per the threat model.

  - `/workspace/.env` (if API keys are ever re-added)

  - `data/cache/` (deletion forces narrator re-runs ~$10+)
  - `git push` to wherever the remote points

A confused agent ("misreads context, gets stuck in a propose-fail loop
and tries to clean up") is by far the most likely failure mode, not a
malicious one — but the effect is the same. Concrete loss modes:

"asking nicely doesn't reliably stop" an agent — drift toward shortcut
paths is a documented failure mode of long-running sessions. With ~144
tick opportunities over 3 days, the empirical rate of "agent reads
instruction and follows it correctly" is not 100%.

4. High: `torch.save(xattn_state.pt)` is non-atomic; backup_to_external.sh can ship a torn 419 MB checkpoint.

Status: OPEN, high

Details:
`src/train/train_xattn.py:285` writes the trainable state dict directly
via `torch.save(sd, run_dir / "xattn_state.pt")`. The file is ~419 MB
and takes several seconds to write at NVMe speeds. No tmp+rename guard.

`scripts/backup_to_external.sh:116-123` runs every 30 minutes and uses
`aws s3 sync` / `rclone sync` / `huggingface-cli upload` directly on the
run directory. These tools open each source file and stream-read it. If
the trainer is mid-`torch.save` at the moment the sync starts reading
`xattn_state.pt`, the upload contains the prefix of the file as it
existed at that instant — a torn pickle. The remote copy is then
committed; the sync tool does not detect end-of-file truncation.

Concrete worst-case ordering:
  t=0    trainer at step 1450/1500, GPU lock held
  t=20s  backup_to_external.sh cron fires
  t=21s  backup starts streaming xattn_state.pt — file is 0 bytes
  t=45s  trainer reaches line 285 — torch.save begins writing
  t=50s  backup reads partial bytes as torch.save extends the file
  t=58s  backup aws s3 cp finishes; remote receives partial bytes
  → remote `runs/exp_NNN/xattn_state.pt` is corrupt.

Window per write is ~1-2 seconds for 419 MB at ~200 MB/s; backup cadence
is 30 min; ~36 experiments over 36 hours → expected collisions ~1-2.

Recommendation:
Three-line surgical fix in `train_xattn.py:285`:

    sd_path = run_dir / "xattn_state.pt"
    sd_tmp = sd_path.with_suffix(".pt.tmp")
    torch.save(sd, sd_tmp)
    sd_tmp.rename(sd_path)

Optional belt-and-suspenders: in `backup_to_external.sh`, before sync,
check whether `/workspace/.gpu.lock` is held; if it is, skip backup for
this tick (or skip the runs/ subtree). The first fix alone (atomic
rename) is sufficient and lower-risk; the second is hardening.

Verify:
    python3 -c "import torch, time; sd={'k': torch.zeros(100_000_000)}; \
                t0=time.time(); torch.save(sd, '/tmp/x.pt.tmp'); \
                import os; os.rename('/tmp/x.pt.tmp','/tmp/x.pt')"
    # While the .pt.tmp is being written, no `xattn_state.pt` is observable
    # at the final path; the rename is instantaneous.

Specialists contributing: state-auditor.

5. High: Convergence halt is silently disabled forever if any HN-FPR value in the window is None.

Status: OPEN, high

Details:
`run_next_experiment.py:349` reads:

    if all(v is not None for v in worst_values):
        best = min(worst_values)
        first = worst_values[0]
        if (first - best) < thresh:
            return f"convergence: ..."

If any run in `xattn_valid_all[-window:]` carries
`hn_fpr_worst_stripped: None`, the halt block is skipped — but the loop
keeps running. None can appear legitimately:
  - Pre-review-013 rows (no schema migration; older rows lack the field)
  - A run where the decision threshold was NaN (degenerate eval, < ~100
    legit predictions)
  - A run where `predictions_stripped.jsonl` was missing (the
    `summary[mode] = {"missing_predictions": True}` branch in
    `run_post_processing`)
  - A run whose `hard_negative_fpr_at_decision_threshold_1pct` dict was
    empty `{}` (chained from any of the above)

until that row exits — potentially many runs later. Combined with

6. High: Trainer subprocess group is not isolated — timeout-kill leaks DDP workers and dataloader children.

Status: OPEN, high

Details:
`launch_trainer` (run_next_experiment.py:449-452) calls
`subprocess.Popen(cmd, ...)` without `start_new_session=True` (or
`preexec_fn=os.setsid`). When `proc.wait(timeout=timeout_sec)` raises
TimeoutExpired and the code falls through to `proc.send_signal(SIGTERM)`
and later `proc.kill()`, the signal is delivered ONLY to the immediate
accelerate-launch parent. DDP rank workers, dataloader workers, and any
other accelerate-spawned grandchildren do NOT inherit the SIGTERM/SIGKILL.

Empirically on single-H100 single-process trainer this is usually fine
(one worker), but the accelerate config may spawn at least one worker,
and dataloader workers (PyTorch default `num_workers > 0`) are separate
processes. Any of these can keep the GPU busy after the launcher
believes it killed the trainer.

This compounds with finding 2: a wedged worker continues holding the
GPU, but the launcher has already released its lockfile in `finally`.
The next tick acquires the lock, tries to launch a new accelerate, gets
a CUDA OOM or "device in use" error from the orphan, and burns Claude
quota proposing fixes.

Recommendation:
One-line surgical fix at run_next_experiment.py:449 — add
`start_new_session=True` to Popen. Then on timeout, broadcast to the
whole process group:

    import os, signal
    try:
        os.killpg(proc.pid, signal.SIGTERM)
    except ProcessLookupError:
        pass
    try:
        rc = proc.wait(timeout=30)
    except subprocess.TimeoutExpired:
        try:
            os.killpg(proc.pid, signal.SIGKILL)
        except ProcessLookupError:
            pass
        rc = proc.wait()

`start_new_session=True` is critical — without it, `killpg(proc.pid, ...)`
either no-ops or signals the wrong group (the agent's group, which
includes the cron shell and possibly claude itself).

Verify (Python, no GPU required — proves the start_new_session + killpg
pairing actually reaps descendants):

    python3 <<'PY'
    import subprocess, os, signal, time
    p = subprocess.Popen(
        ["bash", "-c", "sleep 30 & sleep 30 & wait"],
        start_new_session=True,
    )
    time.sleep(1)
    children = subprocess.check_output(["pgrep", "-P", str(p.pid)]).decode().split()
    print(f"spawned pgid={p.pid}, sleepers={children}")
    os.killpg(p.pid, signal.SIGTERM)
    p.wait(timeout=5)
    leaked = subprocess.run(
        ["pgrep", "-P", str(p.pid)], capture_output=True
    ).stdout.decode().strip()
    assert not leaked, f"FAIL: orphaned descendants {leaked}"
    print("OK: process group fully reaped")
    PY

Expected: "OK: process group fully reaped". If "FAIL: orphaned
descendants ..." appears, the start_new_session/killpg pairing is broken
and the timeout-kill path will leak DDP workers / dataloader children.

9. Medium: Dedup hash differs between `steps: 1500` and `steps: 1500.0` (and similar int/float pairs).

Status: OPEN, medium

Details:

        if isinstance(o, float) and o.is_integer():
            return int(o)

        if isinstance(o, list):
            return [_normalize(v) for v in o]
        return o

    def canonical_hash(cfg):
        canonical = {k: v for k, v in cfg.items() if k not in ('exp_id','rationale')}
        blob = json.dumps(_normalize(canonical), sort_keys=True).encode()

preferred for the POC.
