Review findings:

1. High: The Blackwell bitsandbytes sanity command fails before it checks anything.

Status: OPEN, blocker for using the new Blackwell pod-side check literally.

Details:
RUNBOOK.md adds a "bitsandbytes Blackwell kernel check" at `RUNBOOK.md:345-346`:

    python3 -c "import bitsandbytes as bnb; print('bnb', bnb.__version__); import bitsandbytes.optim as o; opt = o.PagedAdamW8bit([torch.nn.Parameter(torch.randn(8, device='cuda'))]); print('paged_adamw_8bit instantiates OK')"

That command references `torch.nn.Parameter` and `torch.randn` without importing
torch. The synthetic execution in the verification run stubs bitsandbytes so
the command reaches the relevant name lookup, and it fails with
`NameError: name 'torch' is not defined`. On the pod this will fail the same
way after importing real bitsandbytes. It also only constructs the optimizer;
it does not run an optimizer step, which is the point where CUDA kernel
compatibility errors often surface.

Recommendation:
Replace the one-liner with a small multi-line Python block that imports torch,
constructs a CUDA parameter, assigns a gradient, instantiates
`bnb.optim.PagedAdamW8bit`, calls `opt.step()`, then prints success. For
example, the check should prove at least: torch sees CUDA 12.8+, the device
capability is (10, 0) or (12, 0), bitsandbytes imports, PagedAdamW8bit can
execute a CUDA optimizer step, and no "no kernel image is available" error is
raised.

2. High: Preflight still allows bitsandbytes 0.43 even though Blackwell now requires 0.45+.

Status: OPEN, should be fixed before relying on preflight for a Blackwell pod.

Details:
The patch changes `requirements.txt` to `bitsandbytes>=0.45.0,<0.46` and says
0.45+ is required for Blackwell SM_100 / SM_120 paged optimizer kernels
(`requirements.txt:13-19`). The runtime gate did not change. `check_bitsandbytes`
still defaults to `min_version="0.43.0"` (`scripts/preflight_check.py:110`), and
RUNBOOK.md still says "`paged_adamw_8bit` requires `bitsandbytes>=0.43`" at
`RUNBOOK.md:336`.

That leaves the new Blackwell compatibility claim split across three contracts:
requirements says 0.45+, preflight accepts 0.43+, and the runbook says both
0.43+ and 0.45+ in the same gotcha block. If the pod image already contains an
older bitsandbytes, if someone runs preflight before the requirements install,
or if a failed install leaves the old package importable, the gatekeeper can
pass the exact version the Blackwell patch says is unsafe.

Recommendation:
Raise the preflight minimum to 0.45.0 for all pods, or make it
architecture-aware by reading `torch.cuda.get_device_capability(0)` and
requiring 0.45.0 on Blackwell while preserving a lower Hopper floor only if
there is a reason to do so. Update RUNBOOK.md line 336 to match the executable
check. The version line and preflight must say the same thing.

3. High: The documented non-paged 8-bit fallback is not wired into the trainer factory.

Status: OPEN, recovery recipe is not executable from configs.

Details:
RUNBOOK.md says the last-resort Blackwell fallback is to use "non-paged 8-bit
optimizer (`AdamW8bit` instead of `PagedAdamW8bit`) in the trainer configs"
(`RUNBOOK.md:351`). The trainers do not support that config value. All five
trainers call `build_optimizer(... optimizer_name=train_cfg.get("optimizer",
"paged_adamw_8bit"))`, and `src/train/common.py:227-236` accepts exactly two
values:

    paged_adamw_8bit -> bnb.optim.PagedAdamW8bit(...)
    adamw            -> torch.optim.AdamW(...)

There is no `adamw_8bit`, `AdamW8bit`, or `non_paged_adamw_8bit` path. If
PagedAdamW8bit fails on Blackwell and the maintainer follows the new fallback,
the next trainer launch will fail with `ValueError: unknown optimizer`. The
alternative supported fallback, `adamw`, is full torch AdamW, not the 8-bit
non-paged optimizer the runbook describes.

Recommendation:
Either implement the documented fallback in `build_optimizer` (for example,
`optimizer_name == "adamw_8bit"` mapping to `bnb.optim.AdamW8bit`) and add it
to the allowed trainer-config vocabulary, or change RUNBOOK.md to say the
actual supported fallback is `optimizer: adamw` with full-size optimizer state.
If keeping the 8-bit fallback, add a pod-side smoke check for both
`paged_adamw_8bit` and `adamw_8bit` so the recovery path is tested before it is
needed.

Bottom line:
The Blackwell patch points the runbook in the right direction, but the
guardrails are not yet dependable. Fix the runnable bnb check, align preflight
with the 0.45+ requirement, and either implement or remove the documented
AdamW8bit fallback before using this as the RunPod boot checklist.
