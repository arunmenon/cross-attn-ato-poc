Review findings:

1. Blocker: The RunPod bootstrap path assumes a nested repo layout that does not match this git repository.

Status: OPEN, blocker for pushing this repo and following RUNBOOK.md literally on the VM.

Details:
`git rev-parse --show-toplevel` reports this repository root as
`/Users/arunmenon/projects/Foundation-Science/cross_attn_ato_poc`. A fresh
`git clone <your-repo-url> repo` of this repo will therefore place
`RUNBOOK.md`, `scripts/`, `src/`, and `requirements.txt` directly under
`/workspace/repo`. The runbook then says `cd repo/cross_attn_ato_poc`
(`RUNBOOK.md:35-43`), which will fail for this repo-as-root clone. The same
nested assumption appears in the Day-2 cron and single-session commands
(`RUNBOOK.md:96-99`) and in `scripts/backup_to_external.sh`'s default
`REPO_ROOT` (`scripts/backup_to_external.sh:18`).

This is exactly the path a user will take when pushing this repo to a RunPod VM.
The first-boot bootstrap should not depend on remembering whether the remote is
a parent monorepo or the `cross_attn_ato_poc` repo itself.

Recommendation:
Pick one RunPod layout and make every command match it. For this repository,
the simplest is:

    cd /workspace
    git clone <your-repo-url> cross_attn_ato_poc
    cd /workspace/cross_attn_ato_poc

Then update the cron examples and `backup_to_external.sh` default
`REPO_ROOT` to `/workspace/cross_attn_ato_poc`, or make them derive the repo
root from the script location. If you intend to clone a parent
Foundation-Science repo instead, say that explicitly in RUNBOOK.md and make the
clone command use that parent remote. Do not leave the bootstrap ambiguous.

2. Blocker: The documented default auto-loop script does not exist.

Status: OPEN, blocker for the default Day-2 cron-driven auto-loop.

Details:
PLAN.md names cron-driven re-invocation as the default loop mode and says
`scripts/agent_tick.sh` runs every N minutes (`PLAN.md:467`). RUNBOOK.md gives
the concrete crontab entry:

    */30 * * * * /workspace/repo/cross_attn_ato_poc/scripts/agent_tick.sh

(`RUNBOOK.md:96-98`). The file is not tracked; the verification command
returned `scripts/agent_tick.sh exists: 1`, where shell exit code 1 means
missing. Without this wrapper, the default loop cannot start after the baselines
and first x-attn smoke. The single long-session fallback exists, but the
runbook explicitly marks cron as default, and restart recovery assumes the loop
can be restarted from the documented path.

Recommendation:
Add `scripts/agent_tick.sh` or change the runbook/plan to make the single
long-session path the actual supported path. If adding the script, keep it
minimal and deterministic: `cd` to the repo root, activate `/workspace/.venv`,
check the launcher lock/halt status, then pipe the fixed RUNBOOK §6 prompt into
the selected CLI (`claude` or `codex`) with logs under `/workspace`. Add a
local shell syntax check and a dry-run mode that prints the command without
launching an agent.

3. High: Preflight does not fail when persistence-critical environment variables are missing.

Status: OPEN, should be fixed before first pod boot.

Details:
RUNBOOK.md requires the pod template to set:

    HF_HOME=/workspace/.hf
    WANDB_DIR=/workspace/.wandb
    TRANSFORMERS_CACHE=/workspace/.hf/transformers
    TOKENIZERS_PARALLELISM=false

(`RUNBOOK.md:25-31`). The same runbook says `HF_HOME=/workspace/.hf` must be
set before importing transformers or the model cache can land on container disk
and disappear on restart (`RUNBOOK.md:228`). `scripts/preflight_check.py`
imports `transformers` during `check_model_download` and
`check_tokenizer_roundtrip` (`scripts/preflight_check.py:152-161`), but it never
checks these environment variables. The probe confirmed `HF_HOME`,
`TRANSFORMERS_CACHE`, `WANDB_DIR`, and `TOKENIZERS_PARALLELISM` are absent from
the preflight source.

This means preflight can pass after caching Qwen artifacts to ephemeral
container disk. That is a RunPod-specific footgun: a restart would lose the
download even though the gatekeeper said the environment was clean.

Recommendation:
Add a `check_persistent_env()` call before any `transformers` import. Fail if
`HF_HOME` is unset or not under `/workspace`, if `TRANSFORMERS_CACHE` is unset
or not under `/workspace`, if `WANDB_DIR` is unset or not under `/workspace`,
or if `TOKENIZERS_PARALLELISM` is not `false`. Also accept `WANDB_MODE=offline`
as already implemented, but still require `WANDB_DIR=/workspace/.wandb` so
offline logs survive restarts. Then update RUNBOOK.md to export these variables
in the bootstrap shell as a fallback even if the pod template missed them.

4. High: The launcher documents per-experiment timeouts but waits forever.

Status: OPEN, should be fixed before unattended H100 runs.

Details:
`src/auto_research/configs/budget.yaml` says per-experiment wall-clock caps are
"enforced by run_next_experiment.py via timeout" and sets 90/150 minute caps
(`src/auto_research/configs/budget.yaml:26-29`). The launcher does not enforce
that. `launch_trainer` starts `accelerate` with `subprocess.Popen` and calls
`proc.wait()` with no timeout (`scripts/run_next_experiment.py:332-351`). The
only timeout-related grep hit in the launcher is the budget comment; there is no
`wait(timeout=...)`, `communicate(timeout=...)`, alarm, or kill path.

On RunPod this is a cost and recovery problem. A hung model download, stuck
dataset reader, CUDA deadlock, or trainer loop that stops printing will hold
the GPU lock and burn wall-clock until a human notices.

Recommendation:
Thread the budget cap into `launch_trainer`: use
`stress_run_max_wall_clock_minutes` when `training.stress_run` is true,
otherwise `max_wall_clock_minutes`. Call `proc.wait(timeout=cap_seconds)`, send
SIGTERM on timeout, then SIGKILL after a short grace period. Write a failure
record with `status="timeout"` and keep the existing lock release in `finally`.
Add a no-GPU test by launching a short Python sleep script with a one-second
timeout and asserting the run is marked failed.

5. High: External backup will not copy the actual x-attn checkpoints.

Status: OPEN, should be fixed before long sweeps.

Details:
The runbook says top-3 x-attn checkpoints live under
`/workspace/checkpoints/xattn_exp_NNN/` and are externally backed up
(`RUNBOOK.md:63-69`, `RUNBOOK.md:208-210`). `train_xattn.py` saves the trainable
state dict to the run directory as `xattn_state.pt`
(`src/train/train_xattn.py:278-282`), and I found no code that promotes top-3
runs into `/workspace/checkpoints/xattn_exp_NNN/`.

The backup script also parses `sweep_state.yaml` incorrectly. `update_sweep_state`
writes `top_3` as a list of dicts with `exp_id` and `auc_stripped`, but
`backup_to_external.sh` prints each dict object directly and then loops over
the whitespace-split result (`scripts/backup_to_external.sh:83-100`). The probe
shows the script would print:

    {'exp_id': 'exp_001', 'auc_stripped': 0.8}
    {'exp_id': 'exp_002', 'auc_stripped': 0.79}

That never maps cleanly to `/workspace/checkpoints/xattn_$exp_id`. Combined
with the path mismatch, the top-3 checkpoint backup is effectively a no-op.

Recommendation:
Pick one checkpoint location. Either back up
`src/auto_research/runs/exp_NNN/xattn_state.pt` directly for top-3 runs, or
teach the launcher/agent to copy promoted top-3 checkpoints to
`/workspace/checkpoints/xattn_exp_NNN/`. Fix the parser to emit
`entry["exp_id"]`, not the whole dict. Add a shell-level dry-run or small Python
helper test with a synthetic `sweep_state.yaml` containing two top-3 entries
and fake checkpoint directories; assert the backup resolves the expected paths.

6. Medium: Fresh clones do not contain the state files the agent instructions require reading.

Status: OPEN, should be fixed before starting the auto-loop.

Details:
`src/auto_research/AGENT_INSTRUCTIONS.md` tells the agent to read
`sweep_state.yaml` and the last five lines of `experiments.jsonl` at every loop
iteration (`src/auto_research/AGENT_INSTRUCTIONS.md:20-24`,
`src/auto_research/AGENT_INSTRUCTIONS.md:44-52`). Both files are ignored and
not tracked:

    .gitignore:22:src/auto_research/experiments.jsonl
    .gitignore:23:src/auto_research/sweep_state.yaml

There is a local ignored `sweep_state.yaml`, but it will not come along when the
repo is pushed or cloned to RunPod. The launcher can create these files after a
run, but the agent playbook asks for them before proposing the first loop
config. A robust agent may infer missing files mean zero history; the written
instructions do not say that.

Recommendation:
Add an explicit bootstrap step before starting the loop:

    python scripts/run_next_experiment.py --halt-check
    python -c "import importlib.util; ...; rne.update_sweep_state()"
    touch src/auto_research/experiments.jsonl

or add a small `scripts/init_auto_research_state.py` that creates empty
`experiments.jsonl` and initial `sweep_state.yaml` if absent. Update
AGENT_INSTRUCTIONS.md so missing state files are treated as "zero completed
runs" rather than an error.

Bottom line:
Do not push-and-run from RUNBOOK.md as-is. Fix findings 1 and 2 before booting
the VM, and fix findings 3-5 before leaving the pod unattended for paid H100
work.
