1. High: The default cross-attention block is far over the planned
   trainable-parameter budget.

Status: FIXED.

Action: src/model/cross_attn_block.py refactored to add a bottleneck:

  Old:  full-H MHA (4×H×H weights) + 4H FFN per block
        → 201M/block at H=4096
        → every_4 (6 blocks) = 1.21B (~6× over the 400M budget)

  New:  cross_dim defaults to H//4 (1024 at H=4096).
        - Down-projections: hidden → cross_dim (q_proj_in, kv_proj_in,
          both bias=False).
        - MHA operates at cross_dim, not hidden_dim.
        - Up-projection: cross_dim → hidden (out_proj, bias=False).
        - FFN at hidden_dim with dim_feedforward = H//2 (2048 at
          H=4096), half the standard transformer 4× ratio.

        out_proj is zero-init so the cross-attn contribution at step 0
        is exactly zero regardless of gate-init choice (belt-and-braces
        with gate_init="zero").

  New defaults preserve the residual-stream dim (so the wrapper can
  drop blocks in as residual-additive layers) while cutting parameter
  cost by ~6×:

        per_block at H=4096 = 33.6M
        every_4 (6 blocks)   = 201.5M
        every_8 (3 blocks)   = 100.8M
        late_only (4 blocks) = 134.4M

  Plus the wrapper's encoder + resampler + kv_projection + LoRA-on-Q
  add ~13M (encoder/resampler ≈ 8M, LoRA-on-Q for 36 layers at r=16 ≈
  4.7M). Total wrapper trainable:

        every_4  : 214.4M  (within 200-400M Stage-1 budget per PLAN.md)
        every_8  : 113.6M
        late_only: 147.2M

  Two new public helpers ship for the trainer to use:
    - estimate_block_param_count(hidden_dim, cross_dim, dim_feedforward,
      n_heads) → int. Pure-Python, no torch needed.
    - estimate_wrapper_trainable_params(insertion_pattern,
      n_hidden_layers, hidden_size, ...) → dict with per-component
      breakdown + total.

  Self-tests at both module levels assert the production-scale budget
  is satisfied:
    - cross_attn_block self-test prints per_block + 3 sweep arms and
      asserts every_4 total ≤ 400M.
    - qwen_xattn_wrapper self-test prints all 3 sweep arms' wrapper
      total and asserts every_4 ≤ 400M.

2. High: The bf16 path is not dtype-safe once delta_t enters the
   resampler.

Status: FIXED.

Action — two layers of defense:

a. src/model/resampler.py forward path: delta_t is now cast to
   encoder_output.dtype BEFORE the sinusoidal time encoding is
   computed, so time_pe has the same dtype as encoder_output, and the
   `kv = encoder_output + time_pe` add stays in the residual-stream
   dtype. The fix is one line inside the inner _PerceiverResampler
   class:

       time_pe = sinusoidal_time_encoding(
           delta_t.to(encoder_output.dtype), hidden_dim,
       )

b. src/model/qwen_xattn_wrapper.py precompute_kv: defensive device +
   dtype handling on entry. Every call reads `next(encoder.parameters()).device`
   and `.dtype`, then:
     - Moves event_type_ids, bucket_ids, event_mask to encoder device
       (kept integer).
     - Moves delta_t to encoder device AND casts to encoder dtype.

   This protects against the common case where the dataloader collate
   produces float32 delta_t and the wrapper has been moved to bf16,
   without requiring the trainer's collate to know about wrapper
   internals.

c. Resampler self-test now includes an explicit bf16 round-trip:
   construct a fp32 encoder_output, cast to bf16, call resampler on it
   with fp32 delta_t, assert output dtype is bf16. Fails loudly if any
   silent fp32 upcast sneaks in.

Verify (pod-side, once torch is available):

    python3 -m src.model.resampler
    # Includes "bf16 dtype safety OK; output dtype = torch.bfloat16"
    # in the success path.
