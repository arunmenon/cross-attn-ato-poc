Review findings:

1. High: The default cross-attention block is far over the planned trainable-parameter budget.

Status: OPEN, blocker for Batch 4 trainer wiring and H100 sweeps unless intentionally accepted.

Details:
`QwenXAttnWrapper` builds one full-size `GatedCrossAttnDense(hidden_dim=hidden_size)`
per insertion layer (`src/model/qwen_xattn_wrapper.py:125-132`). The block's
default FFN is `4 * hidden_dim` (`src/model/cross_attn_block.py:117-118`), and
the block also has full 4096-wide MHA projections. For Qwen3-8B hidden_size
4096, that is about 201.4M parameters per x-attn block. The planned `every_4`
pattern creates 6 blocks, so x-attn blocks alone are about 1.21B trainable
parameters before the encoder, resampler, kv_projection, and LoRA-on-Q. Even
the sparse `every_8` pattern is about 604M. PLAN.md says Stage-1 trainable
params should be about 200-400M, and docs/batch-3-architecture.md focus area 6
expects "hundreds of millions", not a 1.2B default arm.

Recommendation:
Before Batch 4 trainers instantiate this wrapper, choose an explicit x-attn
parameter budget and make the block match it. Reasonable fixes are a
bottlenecked/low-rank cross-attn block, a much smaller FFN, shared x-attn
blocks across insertion layers, or a configurable `dim_feedforward` that is
validated per insertion pattern. Add a param-count smoke that constructs a
fake 36-layer/4096-hidden base and asserts every sweep arm is inside the
accepted budget before launching training.

2. High: The bf16 path is not dtype-safe once `delta_t` enters the resampler.

Status: OPEN, likely to fail or silently upcast on the pod unless the trainer works around it.

Details:
The run configuration is bf16 (`src/auto_research/configs/sweep_space.yaml:35`
and `src/auto_research/experiment_template.yaml:25`). The resampler creates
time encodings in `delta_t`'s dtype (`src/model/resampler.py:51-63`) and then
adds them to `encoder_output` (`src/model/resampler.py:119-120`). A normal
collate path will create `delta_t` as float32. If the wrapper is moved to
bf16, the encoder output and resampler weights are bf16, but `time_pe` remains
float32, so `kv = encoder_output + time_pe` becomes float32. At best this
silently upcasts the structured-side activations in a bf16 run; at worst it
hits dtype mismatches inside LayerNorm/MultiheadAttention when module weights
are bf16. The wrapper also leaves event tensor device/dtype synchronization to
the future caller (`src/model/qwen_xattn_wrapper.py:291-292`), which is fragile
for the first trainer pass.

Recommendation:
Make `precompute_kv` defensive. Move `event_type_ids`, `bucket_ids`,
`delta_t`, and `event_mask` to the wrapper/base device before use; keep ID
tensors integer; cast `delta_t` or `time_pe` to `encoder_output.dtype` before
the add. The minimum resampler-side guard is:

    time_pe = sinusoidal_time_encoding(delta_t.to(encoder_output.dtype), hidden_dim)
    time_pe = time_pe.to(device=encoder_output.device, dtype=encoder_output.dtype)

Then add a pod-side bf16 smoke that constructs the wrapper, calls
`wrapper.to(torch.bfloat16)`, runs a forward pass, and asserts the resampler
K/V and cross-attn inputs stay bf16.

3. Medium: The resampler padding-mask self-test is nondeterministic because dropout is active.

Status: OPEN, test reliability.

Details:
`PerceiverResampler` defaults to `dropout=0.1` (`src/model/resampler.py:135-143`).
The self-test constructs it without overriding dropout (`src/model/resampler.py:194-197`)
and never calls `eval()`. It then compares two separate forward passes for
near-equality after corrupting masked positions (`src/model/resampler.py:210-218`).
With PyTorch installed, dropout can make those two outputs differ even if the
key-padding mask is perfectly correct. That means the first pod-side smoke can
fail for the wrong reason, or pass/fail depending on RNG state.

Recommendation:
Make the self-test deterministic: instantiate with `dropout=0.0`, or call
`resampler.eval()` and wrap the two comparisons in `torch.no_grad()`. If the
intent is to test only masking, dropout should be absent from that check.
After that, strengthen the test to compare corrupted masked positions in both
rows, not only one half-padded row.

4. Medium: The generation-mode limitation is stronger than the docs state.

Status: OPEN, generation/eval ergonomics.

Details:
docs/batch-3-architecture.md says generation would recompute cross-attention
K/V on each `generate()` step (`docs/batch-3-architecture.md:221`), and focus
area 10 asks to verify that. The current wrapper is a plain `nn.Module` with a
`forward()` method (`src/model/qwen_xattn_wrapper.py:193-215`); it does not
expose `generate()`, does not subclass/forward GenerationMixin, and does not
delegate to `base.generate`. So `wrapper.generate(...)` is not merely
inefficient today; it is unsupported. If Batch 4 scoring uses forward log-prob
only, this is not a training blocker. If any path calls HF generation through
the wrapper, it will fail before the K/V caching issue is reached.

Recommendation:
Either document generation as unsupported until a later batch, or implement an
explicit generation API before anything calls it. A clean API would precompute
and hold structured K/V for the duration of generation, delegate to the base
generation path, and clear the cache in a `finally` block. Add a mock-base test
that proves the public behavior: either `generate` intentionally raises a clear
NotImplementedError, or it works and only precomputes K/V once.
