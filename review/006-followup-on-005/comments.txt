Review findings:

1. High: `gate_init="zero"` now makes the cross-attention branch permanently dead.

Status: OPEN, blocker for the zero-gate sweep arm.

Details:
The review-005 fix added a bottleneck and zero-initialized
`self.out_proj.weight` in every `GatedCrossAttnDense`
(`src/model/cross_attn_block.py:151-155`). That is not just a
"belt-and-braces" identity guard. Combined with `gate_init="zero"`, it removes
all first-step gradient paths into the attention branch:

    attn_out = out_proj(attn_out_d) = 0 because out_proj.weight == 0
    gate_a = tanh(alpha_attn) = 0 because alpha_attn == 0
    h = h + gate_a * attn_out = h

Then `d loss / d alpha_attn` is proportional to `attn_out`, which is zero,
and `d loss / d out_proj.weight` is proportional to `gate_a`, which is also
zero. q_proj_in, kv_proj_in, and the bottleneck MHA are behind the same dead
product. The FFN gate can still learn because its FFN output is not zero, but
that path has no structured K/V signal. This invalidates the planned
Round-2 `gate_init=zero` sensitivity runs in `AGENT_INSTRUCTIONS.md`; those
runs will train an x-attn wrapper whose actual cross-attention branch never
opens.

The current self-test does not catch this. It only checks that gradient
objects are non-None for a `small_0.01` block (`src/model/cross_attn_block.py:307-313`),
not that `alpha_attn.grad.abs().sum()` or `out_proj.weight.grad.abs().sum()`
is nonzero for the zero-gate configuration. A zero tensor gradient still
passes `is not None`.

Recommendation:
Remove the zero-init on `out_proj.weight` and rely on the gate for exact
identity, which is the Flamingo-style mechanism. With `alpha_attn=0` and a
normally initialized attention output, the first backward pass gives
`alpha_attn` a nonzero gradient, so the gate can open. Add a pod-side test for
`gate_init="zero"`:

    block = GatedCrossAttnDense(hidden_dim=32, cross_dim=8, n_heads=4,
                                dim_feedforward=16, gate_init="zero")
    out = block(h, kv, kv_mask)
    assert torch.allclose(out, h, atol=1e-6)
    loss = out.sum()
    loss.backward()
    assert block.alpha_attn.grad.abs().sum() > 0

If you choose to keep zero-init out_proj, remove `zero` from the active sweep
space and document that only `small_0.01` can train the attention branch.

2. High: The bf16 fix preserves dtype by sacrificing timing resolution.

Status: OPEN, should be fixed before H100 training.

Details:
The resampler now casts `delta_t` to `encoder_output.dtype` before computing
the sinusoidal time encoding (`src/model/resampler.py:128-130`). In production
that dtype is bf16. This avoids a float32 upcast, but it makes the cumulative
time calculation and sin/cos arguments bf16. That is too coarse for the plan's
timing signal. docs/batch-3-architecture.md says the time base was chosen to
preserve sub-minute timing while still spanning session-scale durations. The
rounding check above shows why bf16 cumsum breaks that: around 8k-30k seconds,
30-60 second changes often round to the same value before the sinusoidal
encoding is computed.

The new self-test only asserts that the output dtype is bf16
(`src/model/resampler.py:247-265`). It does not assert that the bf16 path
retains enough timing information or stays close to a float32 time-encoding
reference.

Recommendation:
Compute the time encoding in float32 on the same device, then cast the
finished positional encoding to the residual dtype before the add:

    time_pe = sinusoidal_time_encoding(
        delta_t.to(device=encoder_output.device, dtype=torch.float32),
        hidden_dim,
    ).to(dtype=encoder_output.dtype)
    kv = encoder_output + time_pe

This keeps the mixed-precision residual stream bf16 while preserving timing
resolution during cumsum and trig. Add a test that compares the fp32 reference
PE for cumulative times separated by 30 and 60 seconds at 10k and 30k seconds,
then checks the final cast does not collapse the rows before attention.

3. Medium: The small-gate signal self-test no longer verifies that the block output changes.

Status: OPEN, test coverage gap.

Details:
Before the review-005 fix, the cross-attn block self-test asserted
`small_0.01` was not identity. The new test removed that assertion and now
only checks the gate magnitudes (`src/model/cross_attn_block.py:291-305`).
The comment still says "small_0.01-gate produces signal", but gate magnitude
alone is not signal. Because `out_proj` starts at zero, the attention residual
is exactly zero on the first forward even with `small_0.01`; only the FFN
branch can move the output. If the FFN branch later regresses, this test would
still pass while the block is identity.

Recommendation:
Restore an output-level assertion for the small-gate case:

    assert not torch.allclose(out2, h, atol=1e-6)

Then add separate gradient-value assertions for attention and FFN subpaths so
the test catches dead branches rather than only allocated gradient tensors.

Bottom line:
Review 005 fixed the parameter count on paper, but the current fix makes the
zero-gate cross-attention arm non-trainable and computes timing PE in bf16
where session-scale deltas lose the resolution this POC depends on. Those two
should be corrected before Batch 4 trainer work or pod sweeps proceed.
