Review findings:

1. Blocker: Stage-0 freezes the new token embeddings that PLAN.md says it trains.

Status: OPEN, blocker for the first pod-side Stage-0 run and every downstream
merged-base arm.

Details:
PLAN.md says Stage 0 trains "frozen base + new-token embeddings + LoRA on
attention+MLP" (PLAN.md:304), and the Batch 4 briefing describes Stage-0 as
"vocab expansion + LoRA" (docs/batch-4-trainers.md:25). The implementation
resizes the embeddings and initializes every new row to the same mean vector
(`src/tokenizer/custom_tokens.py:204-207`), then immediately freezes every
parameter before applying LoRA (`src/train/train_cpt_light.py:66-73`). The
comment acknowledges that the input embeddings remain frozen at mean init and
"trust[s] the mean-init + LoRA-on-attention" (`src/train/train_cpt_light.py:75-79`).

That is not the plan's Stage-0 objective. If all new custom-token input rows
start identical and stay frozen, the model cannot distinguish
<journey_*>, <actor_*>, <event_*>, PII, or bucket tokens at the embedding
layer. LoRA can adapt downstream projections, but it receives identical input
vectors for distinct new tokens. The merged `qwen3-8b-cpt-light-merged`
checkpoint then carries that broken vocabulary surface into both the CPT-light
baseline and Stage-1 x-attn runs. The same freeze pattern also exists in
`train_lora_text_only.py:51-62` and the defensive resize path in
`train_structured_as_text.py:130-144`.

Recommendation:
Make the Stage-0 embedding contract explicit and tested. Train only the newly
added token rows while keeping pre-existing rows frozen, or use a PEFT-supported
trainable-token/module-save mechanism that preserves those rows in the adapter
and merge output. Add a startup assertion after optimizer setup that the new
embedding rows have trainable parameters, old rows remain frozen, and a one-batch
backward pass produces nonzero gradient on at least one new-token row. If the
team intentionally wants frozen mean-init custom tokens, change PLAN.md and the
Batch 4 docs first; otherwise the current code contradicts the core Stage-0
design.

2. Blocker: The launcher overwrites x-attn gate metrics with parser output that cannot parse the trainer's gate logs.

Status: OPEN, blocker for the auto-research loop's halt and run-record
contracts.

Details:
`train_xattn.py` records the real gate trajectory every step and writes
`gate_trajectory.json` atomically (`src/train/train_xattn.py:220-239`). It also
writes `metrics.json` with `max_gate_magnitude` computed from that trajectory
(`src/train/train_xattn.py:241-276`). Immediately after a successful trainer
run, `run_next_experiment.py` invokes `scripts/parse_metrics.py` as step 1 of
post-processing (`scripts/run_next_experiment.py:540-543`,
`scripts/run_next_experiment.py:343-348`). The parser always writes its own
`metrics.json` and `gate_trajectory.json` (`scripts/parse_metrics.py:117-118`),
overwriting the trainer's authoritative files.

The parser is also looking for a different log format. It only recognizes
`gate_magnitude...` (`scripts/parse_metrics.py:33`) and HF-style quoted loss
records (`scripts/parse_metrics.py:29-32`), while `train_xattn.py` prints
`step ... loss ... max_gate ...` (`src/train/train_xattn.py:230-232`). The
verification above shows the string contract mismatch. The result is that a
successful x-attn run can finish with the trainer's real `max_gate_magnitude`
and trajectory on disk, then the launcher replaces them with `None` and an empty
trajectory. `experiments.jsonl` reads `max_gate_magnitude` from the overwritten
`metrics.json` (`scripts/run_next_experiment.py:552-567`), and the zero-gate
halt rule treats missing values as zero (`scripts/run_next_experiment.py:239-247`).

Recommendation:
Choose one owner for run-level trainer metrics. The simplest fix is: trainer
metrics are canonical on success; `parse_metrics.py` may only fill missing
fields or failure diagnostics, never overwrite `metrics.json` or
`gate_trajectory.json` after a successful trainer wrote them. If parser output
must remain canonical, update it to parse the actual `step ... loss ...
max_gate ...` format and to preserve the richer per-layer trajectory already
written by `train_xattn.py`. Add a no-GPU unit test with a fake run directory
that contains trainer-written `metrics.json`, `gate_trajectory.json`, and a
`train.log` with `max_gate`; after `run_post_processing`, assert the original
`max_gate_magnitude` and trajectory survive.

3. High: The shared config template routes raw-Qwen arms to the merged CPT checkpoint.

Status: OPEN, should be fixed before launching the arm matrix.

Details:
The shared experiment template sets `training.base_checkpoint` to
`/workspace/checkpoints/qwen3-8b-cpt-light-merged` for every arm
(`src/auto_research/experiment_template.yaml:17-20`). `train_cpt_light.py`
and `train_lora_text_only.py` have correct raw-Qwen fallbacks
(`src/train/train_cpt_light.py:60`, `src/train/train_lora_text_only.py:45`),
but those fallbacks are bypassed whenever a generated config includes the
template's `base_checkpoint`.

That breaks two arm contracts. First, Stage-0 CPT-light is supposed to start
from raw Qwen3 and then produce the merged CPT checkpoint; with the template as
written it tries to load the checkpoint it is supposed to create. If that path
does not exist yet, the first Stage-0 run fails before training. If it does
exist from a prior run, Stage-0 recursively trains from an already-CPT-adapted
base. Second, the LoRA-text-only baseline is documented as "raw Qwen3-8B"
(`src/train/train_lora_text_only.py:1-5`, docs/batch-4-trainers.md:24,
PLAN.md:325), but the same template makes it start from the merged CPT base.
That invalidates the baseline comparison.

Recommendation:
Split base checkpoint selection by arm. Either omit `training.base_checkpoint`
from the generic template and let each trainer's default apply, or add explicit
per-arm validation in the launcher:

    cpt_light and lora_text: Qwen/Qwen3-8B unless explicitly overridden
    structured_as_text and xattn: /workspace/checkpoints/qwen3-8b-cpt-light-merged
    event_only: no LM checkpoint required

Fail fast if `arm=cpt_light` points at the Stage-0 merged output path or if
`arm=lora_text` points at a CPT-merged checkpoint without a deliberate override
field.

4. High: Eval scoring silently accepts missing or truncated verdict markers and unverified label-token encodings.

Status: OPEN, should be fixed before trusting any AUC from the LM arms.

Details:
`run_three_mode_eval` is intended to score the next token after
`<risk_verdict>\nlabel:` without exposing the ground-truth label. The code
searches the transformed text for the marker and truncates only if it is found
(`src/train/eval_runner.py:117-120`). If the marker is absent, the full example
continues to tokenization, which can include the ground-truth label and the
rest of the verdict block. If the marker is present but the prefix before it is
longer than `max_length`, tokenizer truncation can remove the marker anyway
(`src/train/eval_runner.py:123-129`). In both cases the scorer still uses the
last non-pad token (`src/train/eval_runner.py:155-163`) and emits a prediction
record, so the failure mode is a plausible-looking AUC from the wrong position.

`get_label_token_ids` has the same silent-assumption problem. It takes the
first token from `tokenizer.encode(" fraud")` and `tokenizer.encode(" legit")`
without asserting that either label is single-token or that the first token IDs
are distinct (`src/train/common.py:129-142`). The Batch 4 briefing explicitly
calls this out as a real-Qwen3 verification requirement.

Recommendation:
Make eval fail closed. Before tokenization, assert every transformed example
contains exactly one marker and truncate to marker-prefix. After tokenization,
assert the decoded token sequence still ends at the marker prefix or use the
existing `find_label_token_position` helper instead of assuming the last real
token is the score position. On startup, assert `tokenizer.encode(" fraud",
add_special_tokens=False)` and `tokenizer.encode(" legit", add_special_tokens=False)`
are non-empty, single-token, and distinct; if Qwen3 makes either label
multi-token, implement explicit multi-token label scoring rather than first-token
scoring.

5. High: The structured-as-text baseline does not use the generator's event serialization.

Status: OPEN, should be fixed before running the structured-as-text baseline.

Details:
The Batch 4 briefing says `_serialize_events_compact` must produce
byte-identical output to `data/gen/build_dataset.py::_event_to_line`. It does
not. The generator emits `<event_txn>t=7 high low` for the sample above, while
the trainer emits a wrapped block with `t=7 event=txn actor=<actor_human> high
low`. The code paths show the same divergence directly:
`data/gen/build_dataset.py:140-155` starts each event with
`<event_{ev['event']}>t={ev['t']}` and never includes actor; the trainer starts
with plain `t=... event=...`, conditionally adds actor, and wraps the entire
sequence in `<events>` tags (`src/train/train_structured_as_text.py:51-75`).

This makes the load-bearing apples-to-apples baseline train and evaluate on a
different event-token distribution than the text corpus and prompt references.
It also introduces `event=txn` plain-text tokens that are not part of the
custom event-token vocabulary.

Recommendation:
Reuse `_event_to_line` directly or move the event-line serializer into a shared
module imported by both the data generator and the trainer. Add a small
torch-free test that builds one representative event dict and asserts the
structured-as-text preamble lines exactly match the generator output for that
same event sequence. If the extra actor line or `<events>` wrapper is desired,
document it as an intentional baseline variant instead of claiming byte
identity.

Bottom line:
Do not start the pod-side Path A run matrix until findings 1 and 2 are fixed.
They would otherwise produce either a broken Stage-0 vocabulary adaptation or
launcher records that erase the x-attn gate evidence the sweep depends on.
